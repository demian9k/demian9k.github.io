**스크럼**\
애자일 개발방법론 중 하나

**스프린트**\
팀이 일정량의 작업을 완료하는 시간이 정해진 짧은 기간(1~4주)

**스크럼 팀**\
하나의 서비스를 만들기 위한 architect, domain expert, software, QA, design 등 구성원 모임
스크럼 마스터가 팀의 리더로 협업을 촉진하고 팀의 장애물을 제거한다.

**스크럼 미팅 Daily Scrum**\
매일 아침 스탠드업 미팅을 통해 현재 하고 있는 일을 스크럼 팀과 공유

**스프린트 계획 Sprint Planning Meeting**\
요구사항을 backlog(일감 목록)로 남긴다. 스프린트 횟수를 결정하고 backlog를 각 스프린트에 배분한다.\
스프린트가 끝나면 백로그 완료 일감을 기준으로 팀을 생산성을 측정한다. 이것을 속도라고 한다.\
이 속도를 기준으로 팀의 스프린트 계획이 세워진다.
continuous planning 이라고 부르는 절차로 전통적인 폭포수 모델에 비해 상황에 따라 적응적인 계획이다.\
**스크럼에서는 프로젝트의 일정, 비용, 품질이 고정**시키고 각 진행상황에 따라 일의 범위가 조정된다.\
반면 기존의 전통적인 개발프로세스는 **범위는 고정되어 있고, 일정, 비용, 품질이 가변적**이다.

**평가 Sprint Review**\
시연 및 검토하여 백로그들이 모두 구현되고 요건을 만족하는지 확인한다. 피드백을 받아 다음 스프린트에 반영한다.

**회고 Sprint Retrospectives**\
스크럼 팀이 프로젝트를 돌아보는 과정이다. 좋았던 점과 안 좋았던 점을 논의하고 개선점을 찾아 다음 스프린트에 적용할 수 있다.

<BlogComponents.FontSizeStrongTitle fontSize={'1.5em'}>
  {' '}
  5.1.2 DDD의 Bounded Context 를 사용한 도출
</BlogComponents.FontSizeStrongTitle>

MSA에서 database per service 에 따라 각 데이터는 직접 접근하지 않는다.
그래서 전체 데이터 모델 중에 어떤 데이터를 각 MSVC 가 소유해야 하는지를 식별하는 것이 중요하다.

DDD의 전략적 설계에서 bounded context는 비즈니스 응집성으로 구분한 개념으로 단일 마이크로서비스로 식별하는 단위가 된다.

Bounded Context를 사용해서 설계하면, 데이터를 기능과 분리하지 않고
sub domain에 속한 도메인 모델로 정의한다.

아래 그림엔 서브 도메인은 보이지만 그에 속한 도메인 모델은 보이지 않는다.
하지만 서브도메인과 bounded context의 관계, 그리고 어떻게 마이크로서비스로 도출되는지 이해하는 데 도움이 된다.

**마이크로서비스의 적정 크기**\
논란이 많은 주제이지만 일반적으로 코드의 라인 수 같은 양으로 판단하지 않고 업무 맥락에서 정성적으로 판단한다.
이 판단 기준은 자율적으로 수행 가능한 업무의 단위, 개념의 일관성, 기능의 응집성 등을 말하는 것이다.
이것은 비즈니스 도메인, 조직의 성숙도에 따라 상대적으로 달라진다.

<BlogComponents.FontSizeStrongTitle fontSize={'1.8em'}>
  5.2 DDD로 설계하기
</BlogComponents.FontSizeStrongTitle>

클라우드에서의 MSA 적용을 통해 얻을 수 있는 독립적 개선과 배포, 장애 격리 및 장애 발생시 빠른 대응을 위해
마이크로서비스를 응집성 있게 식별하는 것이 중요하다.

전략적 설계는 마이크로서비스의 내부 구조를 설계하고, 상세한 내용은 전술적 설계법을 사용한다.
이 방법들은 빠른 기술변화에도 유연한 애플리케이션을 설계할 수 있도록 돕는다.

<BlogComponents.FontSizeStrongTitle fontSize={'1.8em'}>
  5.3 DDD의 전략적 설계
</BlogComponents.FontSizeStrongTitle>
DDD는 하나의 큰 도메인(전체 애플리케이션) 을 전략적으로 중요한 것들을 찾아 도메인을 여러개로 나누고 각
도메인별로 개별 기능을 해결하는 방법을 기본으로 삼는다.

각 나눠진 도메인들은 논리적으로 구분되는 여러개의 하위 영역으로 서브도메인이라고 부른다.

서브 도메인로의 분할은 소프트웨어에서 어떤 문제를 해결하기 위한 영역으로 하나의 큰 문제를 여러개의 작은 문제로 나눠서 해결하는 접근법으로 Divide and Conquer approach 와 그 맥락을 같이 한다.

서브 도메인은 3가지 유형으로 나뉜다.

<BlogComponents.FontSizeStrongTitle fontSize={'1.3em'}>
  핵심 서브도메인(Core Domain)
</BlogComponents.FontSizeStrongTitle>
다른 경쟁기업과 차별화를 만들 비즈니스 영역으로 기업의 프로젝트에서 여러 관심사 중 높은 우선순위를 갖고,
소프트웨어 개발에서도 전략적으로 가장 큰 투자가 필요한 영역이다.

ex) 아마존의 추천시스템, 쿠팡의 로켓배송 등

<BlogComponents.FontSizeStrongTitle fontSize={'1.3em'}>
  지원 서브도메인(Supporting Subdomain)
</BlogComponents.FontSizeStrongTitle>
비즈니스에 필수적이지만 핵심은 아닌 부분으로 볼 수 있다. 그러나 핵심 도메인을 성공시키기 위해서 반드시
필요한 영역으로 핵심 도메인에 이어 2번째로 중요하다.

<BlogComponents.FontSizeStrongTitle fontSize={'1.3em'}>
  일반 서브도메인(Generic Subdomain)
</BlogComponents.FontSizeStrongTitle>
비즈니스적으로 특별하진 않지만 전체 비즈니스 솔루션에 필요한 부분으로 기존 제품으로 구매해서 대체할 수
있는 영역이다.

<BlogComponents.FontSizeStrongTitle fontSize={'1.3em'}>
  Ubiquitous Language, bounded context
</BlogComponents.FontSizeStrongTitle>
고객이나 현업 담당자가 사용하는 비즈니스 언어와 소프트웨어 개발 과정에서 사용하는 언어가 같은 의미에도
다르게 사용되는 경우가 많다. 그리고 같은 발음의 단어라도 특정한 맥락에 따라 의미가 다르게 해석될 수 있는
경우도 많다.

DDD 에서는 언어의 이러한 사용으로 인해 발생하는 문제를 방지하기 위해,
특정 도메인에서 의도를 명확히 반영하고 핵심 개념을 잘 전달 할 수 있는 언어를 유비쿼터스 언어로 정의한다.

유비쿼터스 언어는 일반적으로 데이터모델링에서 사용하는 **표준 단어/용어 사전과 다른 개념**으로
세부적인 도메인에서 특정 업무와 관련된 사람들 사이에서 자율적으로 정의되고 통용되는 언어이다.

예를 들어 결제 도메인에서의 고객, 배송 도메인에서의 고객은 의미가 다를 수 있다.
결제 도메인에서 고객은 금전을 지급하고 구매한 결제자(payer)이고 배송 도메인에서는 물건을 받을 수취인(recipient) 으로 각각 다르게 정의할 수 있다.

도메인 모델은 특정 비즈니스 맥락에서 통용되는 개념들의 관계를 정의한 모형이다. 그래서 잘 설계된 도메인 모델을 보면 비즈니스를 이해할 수 있다.

여러가지 도메인 모델을 설계하다 보면 경계가 생긴다. **유비쿼터스 언어들의 개념이 상이한 경계**가 도메인의 경계인 `Bounded Context`이다.

도메인 모델은 도메인과 관련된 업무를 수행하는 제품 책임자, 도메인 전문가, 개발자를 비롯한 구성원들이 업무를 이해하는 기본 모형이 된다.
도메인 모델에 정의된 언어로 업무 협의를 진행하고, 개념을 발전시켜 나간다.
또한 이 모델의 언어를 그대로 사용해 설계 산출물을 표현하고 소스코드에서도 그대로 사용한다.

새로운 구성원이 팀에 합류하더라도 소스코드의 도메인 모델로 도메인 전문가와 무리없이 소통할 수 있다면 도메인과 서비스에 대한 비전을 공유할 수 있고
서비스는 지속적으로 민첩하게 개발, 유지될 수 있다.

유비쿼터스 언어를 바운디드 컨텍스트, 도메인 모델의 관계를 고려하여 다음과 같이 정의할 수 있다.

> 바운디드 컨텍스트 내의 도메인 주요 개념을 표현하기 위해 도메인 내에서 공통으로 사용하는 언어

<BlogComponents.FontSizeStrongTitle fontSize={'1.3em'} fontColor={'#525252'}>
  Context Mapping
</BlogComponents.FontSizeStrongTitle>
`Bounded Context` 를 식별할 때 컨텍스트는 내부적으로 응집성이 높고 다른 컨텍스트와 의존관계가 낮아야
한다는 원칙 아래에서 설계한다.

그것은 컨텍스트 사이에 아무런 관계가 없다는 의미는 아니다.
여러개의 바운디드 컨텍스트로 식별하면 비즈니스 로직 연결을 위해 컨텍스트간에 연결되어야 하는 경우가
발생한다.

이러한 연결하는 것을 DDD의 `Context Mapping` 이라고 부르며 두 컨텍스트 사이의 선을 그려서 표시한다.
컨텍스트 사이 매핑 관계를 표시한 다이어그램을 컨텍스트 맵이라고 한다.
컨텍스트 맵은 다수의 매핑으로 구성되며 여러가지 매핑 관계가 있다.

**소비자와 공급자(Customer-Supplier)**

공급하는 컨텍스트는 upstream(상류) 로 소비하는 컨텍스트는 downstream(하류) 로 표시한다.
데이터의 흐름은 상류에서 하류로 흐른다.
반대는 가능하지 않으며, 상류의 변화가 생기면 하류에서 변화를 따른다.
공급자는 소비자가 원하는 기능을 제공해야 한다.

**준수자(Conformist)**

준수자는 소비자와 공급자 매핑에서 상류 팀이 하류팀의 요구를 지원하지 않거나 못하는 경우에 이 매핑을 사용한다.
하류팀은 상류 팀이 제공하는 모델을 그대로 사용하는 것이다.

**충돌 방지 계층(Anti-Corruption Layer)**

하류 팀이 상류팀의 모델에 영향을 받을 때 하류 팀의 고유 모델을 지키기 위한 번역 계층을 만드는 것이다.
번역 계층은 상류 모델의 변경 없이 하위 모델과 통합하기 위해 데이터를 변환하는 것은 구현한 것이다.

레거시 시스템을 변경 없이 MSVC 아키텍처로 통합할 때 유용하게 사용할 수 있다.

**5. 액터 찾기**

커맨드를 찾고 나면 커맨드를 실행하는 액터(actor) 를 도출한다.
user, organization, role 등을 의미하며 추상적으로 식별하지 않고 구체적을 역할을 도출해야 한다.
회원이나 관리자 같은 개념보다는 판매자, 구매자, 상품관리자, 배송관리자 와 같이 구체적으로 도출한다.
액터를 도출하면서 이전에 식별되지 않은 커맨드나 도메인 이벤트가 추가로 나오면 추가할 수 있다.

액터는 노란색의 작은 스티커를 사용해 커맨드 왼쪽 아래에 붙여 액터가 커맨드를 조작한다는 것을 명시적으로 표현한다.

액터를 도출하면 다음과 같은 문장으로 커맨드, 이벤트를 검토할 수 있다.

- 판매자가 상품을 등록하면 \[상품등록됨\] 이라는 이벤트가 발생함으로서 시스템이 동작한다.
- 구매자가 상품 주문을 취소하면 \[상품주문취소됨\] 이벤트가 발생하고, 이어서 주문 취소된 상품의 재고를 변경하는 \[재고변경됨\] 이벤트가 발생하여 시스템이 동작한다.

이 문장이 자연스럽지 않다면 커맨드와 도메인 이벤트를 변경하거나 새로 도출해야 한다.

이 단계까지 완료되면 전체 시스템의 큰 그림을 시각화하여 생각하지 못했던 것들을 알 수 있게 되며, 쉽게 공유된다.

액터와 마찬가지로 구체적인 표현으로 도출하면 좋은데 이유는 애그리거트를 구체적으로 식별할수록 컨텍스트의 경계를 식별하는 데 유용하기 때문이다.

**7. 바운디드 컨텍스트 정의하기**

애그리거트 이름에서 이 요소들이 비슷한 지 확인할 수 있다.
이름이 같거나 유사한 애그리거트를 완전히 다른 애그리거트와 구분해서 경계를 그린다.
이전에 식별했던 도메인 이벤트, 커맨드, 액터, 애그리거트를 모두 고려해 경계를 식별한다.

식별한 경계는 바운디드 컨텍스트가 된다.
각 바운디드 컨텍스트에는 이름을 붙이는데 보통 컨텍스트 내부 애그리트 이름으로 한다.
컨텍스트 내에 복수개의 애그리거트가 있는 경우 전체를 대표하는 이름으로 선정한다.

애그리거트가 식별된 결과를 보면 동일한 이름의 애그리거트가 멀리 떨어져 있을 수 있는데
이 경우 큰 애그리거트가 있는 곳으로 옮길수 있다.

다시 말해 모든 관련 요소들을 애그리거트 중심으로 모듈화해야 한다는 것을 의미한다.

2. 구매 컨텍스트에 \[주문\], \[결제\], \[주문아이템\]이 애그리거트로 식별됐지만
   주문할 아이템을 검색한 후 주문에서 결제까지 하나의 흐름으로 이어지기에 분리하지 않고 \[구매\] 컨텍스트로 묶어서 정의했다.

**정책 도출 및 연관관계 정의** \
정책은 이벤트 뒤에 따라오는 반응적인 비즈니스 로직으로 해당 커맨드를 동작하게 한다.

**WHEN \[도메인 이벤트\] THEN EXECUTE \[커맨드\]**

정책은 도메인 이벤트와 커맨드 사이에 존재한다
정책이 호출하는 커맨드는 제한되지 않고 어떤 컨텍스트에던 있을 수 있다.
그러므로 정책을 도출하면 다른 바운디드 컨텍스트와의 관계도 식별할 수 있다.

아래 예제에서는 구매가 완료 or 취소 되는 경우 재고수량이 변경돼야 한다는 정책이 도출되었다.
아래 순서대로 움직인다.

1. 구매 Context에서 \[구입완료\] \[결제취소됨\] 이벤트
2. \[재고변경\] 정책이 실행
3. \[상품재고수정\] 커맨드가 호출
4. \[상품재고수정됨\] 이벤트가 발생.

8. 컨텍스트 매핑하기

이 단계 이전까지 완료하면 대략적인 바운디드 컨텍스트가 식별되어 관계가 파악이 되므로
이 단계에서는 이 관계들을 컨텍스트 맵으로 표현해야 한다.

관계를 확인할 때는 다음을 고려한다.

1. 호출 관계의 방향을 고려한다.
2. 호출 방식이 동기, 비동기 중 어떤것이 필요한지 판단한다.
   - 호출 방식은 데이터의 일관성 측면과 컨텍스트의 가용성 측면을 고려해서 선택한다.
   - 컨텍스트간 일관된 데이터가 필요하면 동기 호출(실선)로 표현한다.
   - 그 외의 경우는 eventual consistency로 충분하므로 비동기 호출(점선)로 표현한다.

예제에서는 아래와 같이 2가지 경우로 처리한다.

- 구매 -> 상품, 회원에 동기방식으로 연동
- 구매 -> 배송 비동기 방식으로 연동

이렇게 도출한 바운디드 컨텍스트는 마이크로서비스 후보가 된다.

최종적으로 마이크로서비스로 정의하려면 다음과 같은 질문을 고려해 판단해본다.

- 비즈니스 프로세스를 수행하기 위한 하나의 맥락의 단위로 구분될 수 있는가? ( 비즈니스 관점 )
- 마이크로서비스별로 분리된 데이터를 정의할 수 있는가? ( 데이터 관점 )
- 하나의 팀이 독립적으로 운영 가능한 단위인가? (운영 조직 관점 )
- 독립적으로 배포 가능한 단위인가? (배포 관점)
- 변경 시 영향을 받는 마이크로서비스가 존재하는가? ( 변경 영향도 관점)
- 도입을 통한 기대효과를 충분히 활용할 수 있는가? ( 클라우드/MSA 도입 목적 관점)

일반적인 객체 모델링 방식은 자유도가 높아 문제 영역을 파고들수록 복잡한 계측 구조를 만들게 될 가능성이 높은데
DDD의 전술적 설계 방법은 객체들의 역할에 따른 유형을 정의하는 방법을 제공한다.

아래 요소들은 도메인 모델을 구성하는 객체 구성 요소를 설명한다.

**Entity**\
다른 엔티티와 구별할 수 있는 식별자를 가진 도메인의 실체 개념을 표현한다.
식별자는 고유하지만 속성 및 상태는 mutable 하다.
도메인에서 개별성이 있는 개념을 엔티티로 식별하며, 고유 식별자와 변화 가능성이 엔티티가 VO를 구분하는 차이점이다.

구매 엔티티는 식별자를 가졌으므로 엔티티이며
구매자, 구매품 등은 식별자가 없으므로 Value Object이다.
VO는 J2EE에서 사용되는 Value Object와 다른 개념으로 J2EE에서 사용된 VO는 DTO로 부르는 것이 적절하다.

**Value Object**\
VO는 속성이 개별적으로 변화하지 않는 개념적 완전성을 모델링한다
개념적 완전성은 값 객체를 구성하는 하나 이상의 특성들이 서로 연관되어 전체 의미를 이루는 것을 말한다.
VO는 개별 속성이 개별적으로 수정되지 않고 전체 객체가 한번에 생성되거나 삭제되는 객체다.

엔티티와 같이 식별자의 차이에 따라 구별되지 않고 ducktyping처럼 모든 속성이 같은 값을 가졌느냐에 따라 동일함이 결정된다.

**Standard Type**\
표준 타입은 대상의 타입을 나타내는 서술적 객체다.
엔티티나 값 객체의 속성을 구분하는 용도로 사용한다.

예를 들어 VO가 전화번호이면 이 전화번호가 집 전화인지, cell phone인지 구분할 때 standard type을 사용할 수 있다.
일반적으로 데이터베이스 모델링에서 코드 값으로 사용된 것과 같은 역할이다.
일반적으로 java 에선 enum으로 정의한다.

**Aggregate**\
엔티티와 값, 표준 타입 객체들로 도메인을 모델링하면 객체간 계층 구조가 만들어진다.

이 객체들을 묶어서 대표하는 객체가 애그리거트다.
애그리거트 하위 객체는 비즈니스 의존 관계를 갖고 있으므로 비즈니스 정합성을 일치시켜야 한다.
그래서 애그리거트는 **트랜잭션의 기본 단위**가 되어 하위 객체들의 정합성을 일치시켜준다.

애그리거트는 내부 엔티티 중 가장 상위 객체를 애그리거트 루트로 정한다.
이 루트 객체는 다른 엔티티와 VO를 대표하여 다른 컨텍스트와 연계할 때 해당 객체를 통해 통신하게 된다.

하나의 컨텍스트 안에 여러개의 애그리거트가 있고 다른 애그리거트르 참조(reference) 해야 한다면
직접 객체 인스턴스르 갖고 있지 않고 **primary 키 같은 식별자를 통해 참조**한다.
직접 참조하는 경우의 다음과 같은 단점이 있기 때문이다.

- 애그리거트 단위의 트랜잭션 처리 어려움
- 의존관계 복잡성

애그리거트 사이의 일관성은 domain event를 통한 eventual consistency로 유지한다.

**도메인 서비스**\
도메인의 비즈니스 로직 처리가 특정 엔티티나 값 객체에 속하지 않을 때 단독 객체를 만들어서 처리하는 것을 말한다.
도메인 서비스는 상태가 없이 행위만 존재하여 다른 엔티티나 값 객체가 가진 상태값을 통해 로직을 처리한다.

**도메인 이벤트**\
이벤트 스토밍의 결과로 도출된 도메인 이벤트 개념의 구현체로서
서비스간 정합성을 일치시키기 위해 단위 애그리거트의 주요 상태값을 담아 전달되도록 모델링한다.
