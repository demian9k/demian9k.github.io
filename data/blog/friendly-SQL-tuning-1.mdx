## 친절한 SQL 튜닝 정리 - 1

### 1장 SQL 처리 과정과 I/O

SQL 실행전 과정

1. 파싱
   SQL문을 트리로 만들고 문법 오류 검사, 참조(존재하는 테이블인지, 권한은 있는지 등) 검사
2. 옵티마이징
   먼저 수집된 시스템 및 오브젝트 통계정보를 바탕으로 다양한 실행계획 생성 후 효율적인 계획 선택
3. row source 생성
   옵티마이징 절차 후 실행가능한 코드로 만드는 단계

hint는 옵티마이저가 실행계획을 만드는 절차에서 사용자가 의도적으로 어떤 부분에는 어떤 방법을 사용하라고 명시하는 부분이다.\
많이 사용할수록 옵티마이저가 실행계획을 만드는 데 제한을 받게 된다.

파싱 과정은 일반적으로 많은 자원을 소모하므로 동일한 문장의 SQL은 캐시 메커니즘을 사용한다.\
실행된 SQL문장이 파싱되어 저장되는 캐시의 이름은 오라클에서 SGA라는 영역의 라이브러리 캐시(V$SQL)라고 한다.\
캐시에 없는 SQL이 파싱되는것을 hard parsing 이라고 하고 캐시에 있는 SQL을 실행하는 것을 soft parsing 이라고 한다.

하드 파싱은 cpu intensive한 절차이고, soft parsing은 상대적으로 cpu를 덜 사용하여 대부분 i/o 작업에만 자원을 사용한다.

파싱과정에는 아래와 같은 정보가 사용된다.

- 테이블, 컬럼, 인덱스 구조에 관한 기본정보
- 오브젝트 통계 : 테이블 통계, 인덱스 통계, 히스토그램 등 컬럼 통계
- 시스템 통계 : CPU 속도 Single Block I/O 속도, Multi Block I/O 속도
- 옵티마이저 관련 파라미터

이러한 라이브러리 캐시 때문에 일반적인 SQL문과 바인드변수를 사용한 SQL문을 실행했을 때 차이가 발생하게 된다.\
책에서는 로그인 정보를 가져오는 쿼리를 일반적인 SQL문으로 아이디 부분만 변경되는 SQL이 라이브러리 캐시에 쌓이게 되지만\
이런 문장들은 같은 SQL로 인식되지 않으므로 캐시에 쌓이는 효과가 없다.

함수, 프로시저, 트리거, 패키지 등은 생성할 때부터 ID를 갖기 때문에 컴파일된 상태로 딕셔너리에 저장되며 실행될 때 라이브러리 캐시에 저장되어 재사용된다.\
그러나 그 외의 일반적인 SQL 문장들은 문장 자체로 ID가 되기 때문에 위에서 예로 든 로그인 SQL 은 다른 것으로 인식되어 캐시가 효과가 없는 것이다.\
그래서 이런 SQL들을 재사용하기 위해서는 달라지는 부분을 다른 쿼리로 파싱하는 것이 아니라 바인드 변수를 사용해 SQL이 동일한 문장으로서 인식되도록 해야 한다.

**36페이지 데이터 저장구조 및 I/O 메커니즘**

SQL이 수행되는 데 소모되는 대부분의 시간은 I/O 작업에서 발생하며 메모리 I/O와 디스크 I/O 중 디스크 I/O가 직접적인 가장 많은 시간을 소모한다.

책에서는 프로세스 생명주기를 소개한다.\
프로세스는 생성되어(new) 종료(terminated) 될 때까지 ready 상태와 running 상태를 \
interrupt(to ready)와 dispatch(to run)절차를 통해 상태의 전이가 반복된다.

running 상태에서 ready 상태로 갈때 주로 I/O 작업을 위해 기다리는 경우가 많다.
CPU는 계속 일할 수 있음에도 I/O가 CPU의 처리속도보다 느리기에 잠시동안 다른 프로세스에 제어권을 넘기는 것이다.

그래서 I/O 작업이 많으면 SQL 성능이 느려진다.\
스토리지 성능이 좋아지고 있지만 CPU보다는 훨씬 느리다. 그러므로 디스크 I/O 는 SQL 성능을 좌우한다.

데이터베이스 저장구조는 아래와 같다

테이블 스페이스 -> 세그먼트 -> 익스텐트 -> 블록 -> 로우(레코드)

각 단계는 아래 단계의 컨테이너다. 테이블 스페이스는 세그먼트의 그룹이며, 세그먼트는 익스텐트의 그룹인 것이다.

테이블 스페이스는 여러개의 데이터파일로 구성된다.

세그먼트는 테이블, 인덱스, 파티션, LOB등 저장공간이 필요한 오브젝트다\
익스텐트는 공간을 확장하는 단위다. 테이블이나 인덱스에 데이터가 추가될 때 해당하는 세그먼트가 공간이 부족하면 해당 오브젝트가 속한 테이블 스페이스로부터 추가로 할당받는다.

또한 익스텐트는 연속된 블록들의 집합이며 여러개의 데이터 블록으로 구성된다.\
공간을 확장하는 단위는 익스텐트이지만 실제로 저장하는 공간은 데이터 블록이다. DB2, SQL Server 는 블록이라는 용어 대신 페이지(page)라는 용어를 사용한다.

하나의 익스텐트와 블록은 하나의 테이블이 독점한다.\
하나의 익스텐트에 속한 블록들은 모두 같은 테이블의 블록이다.\
또한 해당 블록에 저장된 레코드는 모두 같은 테이블에 속한다.

세그먼트에 할당된 모든 익스텐트는 같은 데이터파일에 위치하지는 않는다.\
하나의 테이블 스페이스가 여러개의 데이터 파일로 구성될때 세그먼트를 가능한 여러개의 데이터 파일로 분산해서 저장하면 파일 경합을 줄일 수 있기 때문이다.\
오라클에서는 DBA_EXTENTS 뷰를 통해 세그먼트에 할당된 익스텐트 목록을 조회할 수 있다.\
이 테이블에는 어떤 익스텐트가 가진 속성들, 상위 세그먼트 ID, 테이블 스페이스 이름, 파일 ID, BLOCK ID, 해당 익스텐트의 BLOCK 갯수 등을 알 수 있다.

**DBA -> 데이터 블록 주소**

모든 데이터 블록은 디스크에서 몇번 데이터 파일의 몇번째 블록인지를 나타내는 주소값을 갖는다. 이것을 DBA라고 부른다.

블록은 데이터를 읽고 쓰는 단위이므로 데이터를 읽으려면 DBA를 확인해야 한다.\
index range scan 후 알게 되는 index rowid 는 DBA와 블록 내 순번(row number)로 구성되므로 레코드를 읽을 때 어떤 블록을 읽어야 하는 지 결정할 수 있다.

테이블을 스캔할때는 세그먼트 헤더에 저장된 익스텐트 맵을 이용하는데, 이 맵에는 각 익스텐트의 첫번째 블록의 DBA를 읽어서 스캔한다.\
테이블 스페이스는 여러개의 데이터파일로 구성되고(일대다)\
하나의 세그먼트는 데이터 파일 여러개 분산되어 저장되고(다대다)\
하나의 데이터파일은 여러개의 익스텐트를 갖는다(일대다)

시퀀셜 액세스는 테이블을 읽을 때 논리적, 순차적으로 주변의 블록들을 읽는 방식이다.\
인덱스 스캔의 경우에는 인덱스 리프블록에서 알수 있는 링크드 리스트 구조의 연결을 통해 순차적으로 스캔한다.\
테이블 스캔의 경우에는 세그먼트가 갖고 있는 익스텐트 목록을을 세그먼트 헤더에 맵으로서 갖고 있으며 이 맵에는 각 익스텐트의 첫번째 블록값을 갖고 있다.

그러므로 데이터를 읽으려면 필요한 익스텐트 목록을 읽어 익스텐트 맵을 얻고 첫 블록 뒤에 연속해서 저장된 블록을 순서대로 읽는다.\
랜덤액세스는 이런 연결을 이용하지 않고 레코드 하나를 읽기 위해 한 블록씩 접근하는 것이다.\
SGA에는 버퍼캐시라는 캐시가 있는데 이것은 SQL로 가져온 데이터를 캐시하는 공간이다.\
(SGA의 라이브러리 캐시는 데이터가 아닌 SQL 문장 자체를 캐시한다.)

논리적 블록 I/O는 버퍼캐시 히트로 발생한 총 블록 I/O 이다\
(정확하는 Direct Path I/O + 메모리 I/O 가 논리적 I/O에 속하지만 대부분 메모리 I/O가 논리적 I/O의 대부분을 차지한다.)\
여러번 실행해도 읽어야 하는 블록 수는 동일하다.

물리적 블록 I/O는 디스크에서 발생한 총 블록 I/O를 말한다.\
여러번 실행하면 버퍼캐시 히트에 의해 물리적 블록 I/O는 줄어든다.\
(또한 버퍼캐시는 시간이 지나면서 다른 블록의 히트에 의해 채워지고 버려지므로 물리적 I/O는 그때 그때 달라진다.)

```
버퍼캐시 히트율( 버퍼캐시의 효율 측정하는 지표 )
BCHR = ( 캐시에서 바로 읽은 블록 수 / 읽은 전체 블록 수 ) x 100
= ( 논리적 I/O - 물리적 I/O / 논리적 I/O ) x 100
= ( 1 - (물리적 I/O / 논리적 I/O) ) x 100
```

OLTP 일 경우 일반적으로 99% 히트율을 달성해야 한다.(튜닝 후)

성능을 결정하는 것은 물리적 I/O지만, 물리적 I/O를 줄일수 있는 방법은 메모리 증설을 통한 버퍼캐시 증가 외에는 없다고 책에서는 소개한다. \
물리적 I/O를 외생변수로 보는것이다. 그에 반해 논리적 I/O는 통제가능한 변수로 보고 있다. \
결론적으로 전체적으로 읽는 블록 갯수를 줄이는 것, 즉, 논리적 I/O 를 줄여서 디스크 I/O를 줄이는 것이 SQL 튜닝의 목적이라고 말한다.

BCHR을 계산하려면 SQL 트레이스를 사용해 나온 Call 통계 정보에서\
Query + Current 갯수가 논리적 I/O 이고 물리적 I/O는 Disk로 표시되므로 이 데이터를 사용해서 구할 수 있다.

그러나 높은 BCHR이 항상 좋은 성능을 의미하지는 않는다. 일반적으로 같은 블록을 비효율적으로 반복하여 읽는 상황이 발생할 수 있어서 이 경우 BCHR이 높게 표시되므로 주의해야 한다.\

**Single Block I/O, Multi Block I/O**

클라이언트가 I/O Call시 하나의 블록을 요청하는 것이 Single이고 일반적으로 인덱스를 사용했을 때 소량으로 블록을 읽을 때 사용된다.\
많은 데이터를 읽을 때는 여러개의 블록을 한번에 요청하는 MultiBlock I/O가 좋다.

MultiBlock으로 가져올 수 있는 블록의 갯수는 설정( db_file_multiblock_read_count ) 할 수 있지만 OS가 가져올 수 있는 블록의 최대 용량은 정해져 있다(OS 별로 다르지만 1MB이다.)

오라클 레벨 I/O 단위는 1블록에 8KB 이고 블록개수를 128개로 설정하면 OS 레벨 I/O 한계(1MB)를 만큼 가져올 수 있게 된다.

MultiBlock 으로 한번에 가져올 수 있는 블록들은 하나의 익스텐트에 속한 블록을 의미하며, 블록을 가져울 수 있는 한계를 넘지 않더라도 다른 익스텐트까지 바로 읽을 수는 없다.\
예를 들어 읽으려는 A 익스텐트 에 20개의 블록이 있고, B 익스텐트에는 4개의 블록이 있다고 하자.

A,B 익스텐트를 모두 읽어야 할때 db_file_multiblock_read_count가 8이라면 두번의 I/O Call을 통해 16개를 가져올 수 있고\
세번째 I/O Call을 통해 A 익스텐트에서 남은 4개의 블록을 가져올 수 있으며 여유공간이 있으므로\
B 익스텐트의 4개의 블록을 더 담을 수 있지만 지금 읽고 있는 A 익스텐트를 경계를 바로 넘어서 B 익스텐트의 4개의 블록을 읽을 수는 없다.

Index Range Scan은 Random Access와 Single Block I/O를 사용하기에 다수의 레코드를 읽을 때 성능에 치명적이다.\
이 방법으로 한 블록에 속한 레코드들을 읽어야 한다면 그 블록을 반복적으로 읽어야 하는 것이다 일반적인 온라인 성격의 SQL을 처리할 때 사용한다.

Table Full Scan은 Sequential Access 와 Multiblock I/O 방식으로 디스크를 읽는다.\
일반적인 배치 성격의 SQL을 처리할 때 사용한다.\
그러나 수십 ~ 수백건의 소량 데이터를 처리할 때 수백만 ~ 수천만 건 데이터를 읽는 것을 비효율적이다.

**버퍼캐시 탐색 메커니즘**

Direct Path I/O를 제외한 모든 블록 I/O는 버퍼캐시를 경유한다.

버퍼캐시는 해시체인 구조로 관리된다.\
블록 번호로 해시 함수로 계산하여 같은 해시 체인에 메모리에 적재한 블록의 포인터를 연결 해놓고 블록이 필요할 때마다 이 해시를 참조하여 사용하게 되는 것이다.

버퍼캐시는 SGA 구성요소이므로 DB를 사용하려는 프로세스들 사이에서 공유자원이 된다.
공유자원은 두 개이상의 프로세스 사이에 경합이 발생할 수 있다.
버퍼캐시는 Latch 라는 Lock을 사용해 한번에 하나의 프로세스만 사용할 수 있게 되어있다.
Latch는 해시체인앞에 하나씩 생성된다. 키를 획득한 프로세스만 해당하는 해시체인에 소속된 블록을 읽을 수 있다.

블록 또한 경합을 막기 위한 Buffer Lock이라는 장치를 갖는다.\
체인의 Latch는 하나의 프로세스가 블록을 찾은 뒤 바로 해제해야 하기 때문에 체인 안에 속한 블록에도 Lock이 필요하다. \
Latch는 블록의 정합성을 위한 것이고 Buffer Lock은 데이터 정합성을 위한 것이다.
