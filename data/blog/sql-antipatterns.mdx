**이 문서는 bill karwin의 저서 SQL Antipattern을 요약한 문서입니다.**

이 책은 25가지 SQL 안티패턴들과 그 방법 대신 쓸수 있는 방법들(Best Practice)를 알려준다.

학술적 접근 보다는 실무적인 관점에서 저술되어 실제로 수행한 업무를 반추 할 수 있었다.

하지만 너무 상식적인 것도 안티패턴으로 소개하고 있기 때문에 그런 경우에는 자세히 적지 않았다.

## Part I - Logical Database Design Antipatterns

### 패턴 1. 무단횡단

책에서는 Product 라는 테이블에 그 제품의 담당자 목록(Accounts)을 표현할 때
여러개의 값을 콤마로 구분한 문자열을 사용했다.

이 방법이 나쁜 이유는 데이터베이스는 이런 형태의 컬럼을 사용하도록 만들어지지 않았기 때문이다.
책에서는 select, aggregate, update 등의 쿼리를 보여주며
일반적으로 쿼리하는 방법이 아니라 특별한 방식으로 쿼리해야 하며
일반적으로 쿼리했을 때와 비교하여 여러번의 SQL 문장이 필요함을 말한다.

해결 방법으로는 Account와 Product 사이에 다대다 테이블을 만들어 사용하는 것이다.

이 방법의 유일한 장점은 애플리케이션에서 콤마로 구분된 문자열과 같은 형태의 값을 조작 없이 그대로 사용하는 경우이며
이것은 일종의 반정규화 효과를 일으킨다고 말한다.
(반정규화는 성능 때문에 조인을 통해 테이블을 결합시키지 않기 위해 컬럼을 정규화하지 않는 것을 말한다.)

### 패턴 2. 순진한 트리

테이블이 계층 구조를 만들 때 가장 단순한 방법은 parent_id 같은 같은 테이블의 pk를 표시한 컬럼을 참조하도록 하는 것이다.

이 방법을 책에서는 인접 목록(Adjacency List)라고 부른다.

이 방법은 단계가 늘어갈 때마다 Left Outer Join을 사용해 쿼리 길이가 늘어간다.

노드를 추가하는 것과 노드를 이동하는 것은 쉬워진다.
그러나 트리에서 노드를 삭제하는 것은 복잡해진다.

서브트리 전체를 삭제하려면 FK 조건이 걸려 있으므로 다수의 쿼를 통해 모든 자손을 찾은 후 가장 아래 단계에서부터 차례대로 삭제하면서 올라가야 한다.

책에서 제시하는 대안적인 모델은 아래 3가지이다.

1. path enumeration(경로 열거)

```sql
CREATE TABLE Comments (
comment_id SERIAL PRIMARY KEY,
path VARCHAR(1000),
bug_id BIGINT UNSIGNED NOT NULL,
author BIGINT UNSIGNED NOT NULL,
comment_date DATETIME NOT NULL,
comment TEXT NOT NULL,
FOREIGN KEY (bug_id) REFERENCES Bugs(bug_id),
FOREIGN KEY (author) REFERENCES Accounts(account_id)
);
```

| comment_id | path     | author | comment                             |
| ---------- | -------- | ------ | ----------------------------------- |
| 1          | 1/       | Fran   | What’s the cause of this bug?       |
| 2          | 1/2/     | Ollie  | I think it’s a null pointer.        |
| 3          | 1/2/3/   | Fran   | No, I checked for that.             |
| 4          | 1/4/     | Kukla  | We need to check for invalid input. |
| 5          | 1/4/5/   | Ollie  | Yes, that’s a bug.                  |
| 6          | 1/4/6/   | Fran   | Yes, please add a check             |
| 7          | 1/4/6/7/ | Kukla  | That fixed it                       |

comment_id 7의 조상찾기

```sql
SELECT *
FROM Comments AS c
WHERE '1/4/6/7/' LIKE c.path || '%';
```

comment_id 4의 자손찾기

```sql
SELECT *
FROM Comments AS c
WHERE c.path LIKE '1/4/' || '%';
```

2. Nested Sets

```sql
CREATE TABLE Comments (
comment_id SERIAL PRIMARY KEY,
nsleft INTEGER NOT NULL,
nsright INTEGER NOT NULL,
bug_id BIGINT UNSIGNED NOT NULL,
author BIGINT UNSIGNED NOT NULL,
comment_date DATETIME NOT NULL,
comment TEXT NOT NULL,
FOREIGN KEY (bug_id) REFERENCES Bugs (bug_id),
FOREIGN KEY (author) REFERENCES Accounts(account_id)
);
```

| comment_id | nsleft | nsright | author | comment                             |
| ---------- | ------ | ------- | ------ | ----------------------------------- |
| 1          | 1      | 14      | Fran   | What’s the cause of this bug?       |
| 2          | 2      | 5       | Ollie  | I think it’s a null pointer.        |
| 3          | 3      | 4       | Fran   | No, I checked for that.             |
| 4          | 6      | 13      | Kukla  | We need to check for invalid input. |
| 5          | 7      | 8       | Ollie  | Yes, that’s a bug.                  |
| 6          | 9      | 12      | Fran   | Yes, please add a check             |
| 7          | 10     | 11      | Kukla  | That fixed it                       |

아래 쿼리는 comment 4와 nsleft, nsright 값으로 그 사이의 자손들을 찾을 수 있다.

```sql
SELECT c2.*
FROM Comments AS c1
JOIN Comments as c2
ON c2.nsleft BETWEEN c1.nsleft AND c1.nsright
WHERE c1.comment_id = 4;
```

아래 쿼리는 comment 6과 nsleft, nsright 값으로 그 사이의 조상들을 찾을 수 있다.

```sql
SELECT c2.*
FROM Comments AS c1
JOIN Comments AS c2
ON c1.nsleft BETWEEN c2.nsleft AND c2.nsright
WHERE c1.comment_id = 6;
```

3. Closure Table
   이 방법에서는 path 테이블을 별도로 두어 ancester(조상), descendant(자손) 필드에 연결된 row의 pk를 표기하는 방법으로 연결한다.

```sql
CREATE TABLE Comments (
comment_id SERIAL PRIMARY KEY,
bug_id BIGINT UNSIGNED NOT NULL,
author BIGINT UNSIGNED NOT NULL,
comment_date DATETIME NOT NULL,
comment TEXT NOT NULL,
FOREIGN KEY (bug_id) REFERENCES Bugs(bug_id),
FOREIGN KEY (author) REFERENCES Accounts(account_id)
);

CREATE TABLE TreePaths (
ancestor BIGINT UNSIGNED NOT NULL,
descendant BIGINT UNSIGNED NOT NULL,
PRIMARY KEY(ancestor, descendant),
FOREIGN KEY (ancestor) REFERENCES Comments(comment_id),
FOREIGN KEY (descendant) REFERENCES Comments(comment_id)
);
```

| ancestor | descendant |
| -------- | ---------- |
| 1        | 1          |
| 1        | 2          |
| 1        | 3          |
| 1        | 4          |
| 1        | 5          |
| 1        | 6          |
| 1        | 7          |
| 2        | 2          |
| 2        | 3          |
| 3        | 3          |
| 4        | 4          |
| 4        | 5          |
| 4        | 6          |
| 4        | 7          |
| 5        | 5          |
| 6        | 6          |
| 6        | 7          |
| 7        | 7          |

자손을 구하는 방법

```sql
SELECT c.*
FROM Comments AS c
JOIN TreePaths AS t ON c.comment_id = t.descendant
WHERE t.ancestor = 4;
```

조상을 구하는 방법

```sql
SELECT c.*
FROM Comments AS c
JOIN TreePaths AS t ON c.comment_id = t.ancestor
WHERE t.descendant = 6;

```

새 leafnode를 #5의 자식으로 insert하는 방법

```sql
INSERT INTO TreePaths (ancestor, descendant)
SELECT t.ancestor, 8
FROM TreePaths AS t
WHERE t.descendant = 5
UNION ALL
SELECT 8, 8;
```

#4 row 와 그 전체 subtree를 삭제하는 방법

```sql
DELETE FROM TreePaths
WHERE descendant IN (SELECT descendant
FROM TreePaths
WHERE ancestor = 4);
```

### 패턴 5. entity-attribute-value 테이블

두 테이블이 공통적인 속성을 가지고 있을 때
이런 속성들을 별도의 테이블에 컬럼이 아닌 row로 저장하는 것이다.

Bugs와 FeatureRequest 테이블이 공통적인 컬럼을 갖고 있을 때
이 테이블을 Issues 라고 이름붙인다.

Issues 테이블에는 PK인 Issue_id 컬럼만 존재하며
나머지 컬럼들은 IssueAttributes 라는 테이블을 만들고
이 테이블 컬럼을
issue_id, attr_name, attr_value 로 구성하여
필요한 컬럼명을 attr_name에
그 값을 attr_value로 사용하는 방법이다.

이 방법을 EAV(Entity-Attr-Val) 테이블이라고 부른다.
이런 방법은 단순하지만 여러가지 단점을 가진다.

데이터 정합성을 해친다.

1. NOT NULL 같은 제약조건 사용 불가
2. 데이터 타입 사용 불가( 일반적으로 모든 타입을 지원하기 위해 attr_value 타입에 문자열을 사용하므로
   날짜 값이 잘못된 경우에 유효값 검증을 할 수 없다.)
   3.FK 사용 불가여 속성 범위를 제한할 수 없다.
3. 속성 이름 제한 불가
   이름의 값을 유동적으로 정할 수 있어서 한 가지 의미의 데이터가 여러가지 attr_name을 가질 수 있다(date_reported, report_date)
   이런 점은 사용자를 혼란스럽게 만들수 있다

행을 일반적인 테이블에서 SELECT 한 것처럼 보이게 재구성할때도
각 필요한 attr_name 갯수만큼 LEFT OUTER JOIN을 필요로 하기 때문에
비용이 많이 든다.

책에서는 이런 방법을 사용하려면 RDBMS가 아닌 DB를 사용하라고 조언하고 있으며
그 경우로는 Berkeley DB, 카산드라 , Couch DB , 하둡, 몽고디비, Redis 등을 소개해주고 있다.

이 테이블을 사용하는 목표는 "가변 속성 지원" 이라고 되어 있는데 잘 와닿지 않는다.

하여간 이 목표를 충족하면서 EAV를 사용하지 않는 대안으로는 다음과 같은 방법이 있다.

#### 해법1 서브타입 모델링

아래 방법들은 여러가지 서브타입 모델링 방법들이다.

- 단일 테이블 상속

  1. 두 서브타입(Bugs, FeatureRequest) 의 모든 속성을 담을 수 있는 공통테이블 Issues를 만든다.
  2. 이 테이블에는 두 테이블에서 필요한 모든 컬럼이 선언되어 데이터가 insert 된다.
  3. 어떤 row가 두 테이블 중 어떤 테이블에 속한것인지를 표시한는 issue_type 컬럼이 존재한다.(BUG 또는 FEATURE 라는 값으로 구분)
  4. 그 외에 컬럼에는 공통적으로 사용하는 컬럼들과 각 테이블에서만 사용하는 컬럼들도 존재한다.

- 구체 테이블 상속

  - 가장 단순하게 두 서브타입별로 테이블을 만드는 것이다.
    공통 속성도 각자 갖고 있으며 각 테이블에서만 사용하는 컬럼은 각각 갖고 있다.

- 클래스 테이블 상속

  1. 서브타입에서 공통적인 속성을 Issues 테이블에 선언하고 각 서브타입별 컬럼들은 각자의 테이블에 유지한다.
  2. Issues 테이블과 Bugs, FeatureRequests 테이블은 issue_id로 FK로 연결하여 일대일 관계를 강제한다.

- Semistructured Data
  1. 서브타입의 수가 많을 경우 JSON같은 형식으로 코딩하여 TEXT 또는 CLOB 컬럼으로 저장한다.
  2. 공통적인 속성을 그대로 Issues 테이블에 컬럼으로 정의하여
  3. 서브타입별 컬럼을 attribute 컬럼( TEXT, CLOB ) 타입으로 정의해 JSON 같은 형식으로 저장하는 형태이다.
  4. 어떤 row가 두 테이블 중 어떤 테이블에 속한것인지를 표시한는 issue_type 컬럼이 존재한다.(BUG 또는 FEATURE 라는 값으로 구분)

### 패턴 7. 다형성 연관

Bugs 테이블은 Comment를 가진다.  
Feature 테이블도 Comment를 가진다.

즉, Bugs 와 FeatureRequest는 Comment라는 부모를 가진다.

목표는 여러가지 부모를 참조하는 것이다.

이 경우를 해결하는 안티패턴은 다음과 같다.

1. 기존 comment 테이블에 issue type 컬럼을 만들고
2. 값을 bugs 또는 featureRequest 로 표시하고 bugs 전용 comment 테이블이었다면 가져야 했을 fk 인 bugs id 대신 두 테이블의 id와 연결되는 issue id를 사용한다.
3. 그러나 두 테이블에 fk를 강제할 수 없으므로 bugs 전용 테이블이었을 때와 다르게 issue_id는 fk가 될 수 없다.

이렇게 하면 comment 테이블을 두 테이블에서 공유할 수 있게 된다.

FK가 강제되지 않으므로 데이터 정합성을 보장할 수 없으며
issue_type의 문자열이 어떤 테이블에 대응되는지를 확인하는 메타데이터도 없다.

해결 방법1의 이름은 교차 테이블 생성이다.

공통 Comment 테이블과 각 자식 테이블마다 다대다 해소 테이블을 만드는 것이다.
예를 들어 Bugs와 Comment를 연결하기 위해서는

Bugs --< BugsComment >-- Comment 관계가 된다.

Comment에는 comment_id(PK), issue_id( 각 교차 테이블마다 FK로 참조됨) 이 있고

Comment 테이블엔 issue_id가 fk로 존재하는데,

### 패턴 15 애매한 그룹

목표: 그룹당 최댓값을 가진 행 얻기
안티패턴 : 그룹되지 않은 컬럼 참조

GROUP BY를 사용한 쿼리에서 SELECT-LIST에 포함도니 컬럼은 그룹당 하나의 값을 가져야 한다는 규칙을
단일 값 규칙(Single-value Rule)이라고 한다.

일반적으로 이 규칙을 어길 경우 데이터베이스는 오류를 발생시킨다.

> mySQL과 sqlite는 동작이 덜 엄격하다.
> mySQL에서는 기본적으로 오류가 발생하지 않고 모호한 쿼리가 허용되며, ONLY FULL GROUP SQL 모드를 사용했을 때만 오류가 발생한다고 한다.

이 패턴은 아래의 쿼리같은 상황에서 발생하는 문제들을 얘기하는 것이다.

3가지 경우를 예로 들었다.

**첫번째 문제적(problematic) SQL**

- Bugs: 소프트웨어 회사에서 버그를 관리하는 테이블
- BugsProduct: 제품(Products) 테이블과 다대다 관계 해소를 위한 테이블

```sql
SELECT product_id, MAX(date_reported), bug_id
FROM Bugs JOIN BugsProducts USING(bug_id)
GROUP BY product_id;
```

1. Bugs, BugsProduct를 bug_id로 조인하는 SQL이다.
2. GROUP BY 기준 컬럼은 product_id이며 MAX 함수를 보고된 날짜(date_reported) 컬럼에 적용했다.
3. 그리고 bug_id 컬럼을 추가적으로 출력하도록 SELECT LIST 절에 포함했다.

Question 1
: date_reported값이 동일했을 때 그룹내 최댓값이라면 쿼리에서 어떤 bug_id 값을 보여줘야 하는가?

---

**두번째 문제적 SQL**

```sql
SELECT product_id, MAX(date_reported), MIN(date_reported), bug_id
FROM Bugs JOIN BugsProducts USING(bug_id)
GROUP BY product_id;
```

Question 2
: 두 종류의 집계 함수인 MAX, MIN 함수를 아래와 같이 사용한다면 그룹 안에서 두개와 다른 행에 대응되는데, 이때는 각 그룹에서 어떤 bug_id를 리턴해야 하는가?

---

**세번째 문제적 SQL**

```sql
SELECT product_id, SUM(hours), MIN(date_reported), bug_id
FROM Bugs JOIN BugsProducts USING(bug_id)
GROUP BY product_id;
```

Question 3
: 집계 함수가 리턴하는 값과 매치되는 행이 없는 경우에는 bug id 값을 어떻게 해야 하는가?

AVG, COUNT, SUM과 같은 함수를 사용할 때 이런 경우가 발생한다.

해법으로는 컬럼을 모호하게 나오도록 조치를 취해야 하는 것이다.

여러가지를 제안하고 있다.

1. 모호한 컬럼 제외하기
   제외해도 상관없는 컬럼은 제거하는 것이 가장 간단하다.

2. 상관 서브쿼리(Correlated Subquery) 사용하기

```sql
SELECT bp1.product_id, b1.date_reported, b1.bug_id
FROM Bugs b1 JOIN BugsProducts bp1 USING (bug_id)
WHERE NOT EXISTS
(SELECT * FROM Bugs b2 JOIN BugsProducts bp2 USING (bug_id)
WHERE bp1.product_id = bp2.product_id AND b1.date_reported < b2.date_reported)
```

상관 서브쿼리(Correlated Subquery)는 바깥쪽 쿼리의 각 행에 대해 한번씩 실행되므로 성능상 최적의 방법은 아니다.

3. 유도 테이블(derived table) 사용하기

```sql
SELECT m.product_id, m.latest, b1.bug_id
FROM Bugs b1 JOIN BugsProducts bp1 USING (bug_id)
JOIN ( SELECT bp2.product_id, MAX(b2.date_reported) AS latest
FROM Bugs b2 JOIN BugsProducts bp2 USING (bug_id)
GROUP BY bp2.product_id ) m
ON (bp1.product_id = m.product_id AND b1.date_reported = m.latest)
```

이 경우 서브쿼리 m이 여러개의 latest 를 반환할 수 있다.

그런 것을 방지하려면

```sql
SELECT m.product_id, m.latest, MAX(b1.bug_id) AS latest_bug_id
FROM Bugs b1 JOIN
( SELECT product_id, MAX(date_reported) AS latest
FROM Bugs b2 JOIN BugsProducts USING (bug_id)
GROUP BY product_id ) m
ON (b1.date_reported = m.latest)
GROUP BY m.product_id, m.latest
```

유도 테이블은 상관 서브쿼리(Correlated Subquery)보다 규모가변성(scalability)이 좋은 대안이다.  
상호 연관되지 않으므로 대부분의 DB에서 서브쿼리가 한번만 수행되지만
임시 테이블에 중간 결과 를 저장해야 하므로 성능상 최적의 방법은 아니다.

> 임시 테이블 중간 결과 저장에 대한 역주  
> 실행 계획이 조인으로 풀릴 수(변환) 있으므로 반드시 중간 결과가 임시테이블에 저장되는 것은 아니다.

4. 조인 사용하기

outer join을 사용해 행이 없을 수도 있는 곳에 NULL이 나오게 한다.

```sql
SELECT bp1.product_id, b1.date_reported AS latest, b1.bug_id
FROM Bugs b1 JOIN BugsProducts bp1 ON ( b1.bug_id = bp1.bug_id )
LEFT OUTER JOIN ( Bugs AS b2 JOIN Bugs Products AS bp2 ON ( b2.bug_id = bp2.bug_id) )
ON ( bp1.product_id = bp2.product_id
AND ( b1.date_reported < b2.date_reported OR b1.date_reported = b2.date_reported AND b1.bug_id < b2.bug_id))
WHERE b2.bug IS NULL;
```

책에서는 이 방법을 추천했지만 아래와 같은 말도 덧붙였다.

> "어떤 방법이 다른 방법보다 성능이 좋다고 가정하지 말고 여러 형태의 쿼리에 대해 성능을 측정해 확인해야 한다는 점을 기억하라."

또한 마지막의 SQL Antipatterns Tip에서 아래와 같이 말하고 있다.

> "모호한 쿼리 결과를 피하기 위해 단일 값 규칙을 따라라. "

### 패턴 24 외교적 면책특권

목표 : 관례 따르기

일반적으로 애플리케이션은 아래와 같은 소프트웨어공학적 습관을 사용하려 노력한다.

- VCS를 사용해 소스코드를 관리한다.
- 단위테스트 자동화를 사용한다.
- 애플리케이션의 요구사항과 구현 전략을 기록하기 위해 명세서 문서 코드 주석을 작성한다.

관례를 사용하면 불필요하거나 반복적인 작업을 대폭 줄여주기에 관례 없이 코딩하는 것보다 시간을 절약할 수 있다.  
노련한 개발자들은 편의를 위해 이런 관례를 희생하는 것이 실패의 지름길임을 안다.

패턴 이름은 DB 코드를 작성할 때 이런 관례가 면제된다고 생각하는 경향이 있기 때문이다.

안티패턴은 DB 코드를 애플리케이션과 동일하지 않은 절차로 처리하는 것(SQL을 second citizen 으로 만든다) 이다.

다음은 **안티패턴**을 따르는 지름길이다.

- 새로운 경량 엔지니어링 프로세스 채택
  - 일부를 생략하여 합리적일 수 있지만 중요 관례를 따르지 않겠다는 말을 완곡하게 표현한 것일 뿐이다.
  - DBA 직원들을 개발자로 대우하지 않아 기술교육에서 배제하는 등의 조치를 취하는 것
  - 데이터베이스 스키마 문서를 기록, 유지하지 않는 것

해법 : 초당적 품질 문화 확립

일반적으로 소프트웨어 공학 생명주기에 포함된 다음 3단계를 엄격하게 지키도록 노력한다.

1. 프로젝트 요구사항을 문서로 명확하게 설계
2. 요구사항에 대한 솔루션을 설계하고 개발
3. 솔루션이 요구사항에 맞는지 검토하고 테스트한다.

그 결과로 DB 개발에서도 QA를 달성할 수 있다.

**증거1: 문서화**

- 자체 문서화 코드(주석 등 코드로 문서화를 대체하려는 무언가를 의미한다고 판단됨)는 없다.
- 코드가 읽기 쉽다면 왜 코드라고 부르겠는가?

가독성 높은 코드 및 노련한 프로그래머는 주의 깊은 분석과 실험을 통해 대부분의 코드를 해독할 수 있지만
코드 해독하는데 문서화하지 않는 것보다 인내와 에너지, 시간을 더 요구한다.  
또한 코드는 빠진 기능이나 해결하지 못한 문제는 알려주지 않는다.

### 작성해야 되는 문서와 작성방법

#### ERD

- 가장 기본적인 문서
- 테이블, 컬럼, 뷰를 설명하는 문서

ERD는 각 테이블과 컬럼, 기타 객체의 목적과 사용법을 기술하기에 적절한 형식이 아니기 때문에
테이블, 컬럼, 뷰에 대한 설명하는 문서도 추가적으로 필요하다.
단순히 DDL이나 DESCRIPTION이 아니라 각 테이블의 목적이나 테이블별로 얼마나 많은 행이 예상되는지
테이블에 어떤 쿼리가 들어올 것으로 기대되는지 테이블에 인덱스가 어떤 용도로 사용되는지도 필요하다.

컬럼의 이름과 데이터 타입으로는 컬럼의 값이 무엇을 의미하는지 어렵다.
컬럼의 값들이 어떤 의미를 가지는지, 양을 저장하는 컬럼의 경우 측정 단위가 무엇인지, 컬럼의 NULLABLE 을 결정한 이유는 무엇인지
UNIQUE 제약조건이 있다면 그 이유는 무엇인지 기술해야 한다.

뷰 또한 마찬가지다.

- 뷰를 생성해야 하는 이유는 무엇인지
  - 뷰가 테이블의 복잡한 관계를 추상화하려는 의도인지,
  - 권한이 없는 사용자를 위해 일부 데이터만 노출하기 위한 것인지 등
- 어떤 애플리케이션에서 뷰를 이용할 것으로 기대되는지를 기술해야 한다.

#### 관계

참조 정합성(FK) 제약조건은 테이블간 종속성을 구현한 것이지만 모델링하려는 제약조건의 의도를 모두 알려주지는 않는다.
일부 조건들은 일반적으로 사용되는 목적을 벗어나 사용되어 있다면 그것을 의도를 기술해야 한다.

#### 저장 프로시저

- API 처럼 문서화해야 한다.
  - 프로시저가 어떤 문제를 해결하기 위해 작성되었는지, 입출력 파라미터의 데이터 타입과 그 의미를 기록한다.
  - 특정 형태의 쿼리를 대체해 성능 병목을 제거하려는 의도인지 권한이 없는 사용자를 위해 일부 데이터만 노출하기 위해 사용하는 것인지를 기술한다.

#### SQL 보안

- 애플리케이션을 위한 DB 계정
  - 각 계정이 가지는 접근 권한
  - 어떤 계정이 어떤 역할을 가지는지
  - 백업이나 보고 같은 특정 작업용 계정이 있는지
  - 시스템 수준에서 사용하는 보안 규정은 어떤 것이 있는지(클라이언트가 SSL을 통해 DBMS 서버에 접근하는지 와 같은)
  - 무차별 대입공격과 같은 불법 인증시도를 발견하고 차단하기 위해 어떤 방법을 취하고 잇는지
  - SQL 인젝션 취약점을 예방하기 위해 전반적 코드 검토를 실행했는지를 기술한다.

#### 데이터베이스 기반 구조.

- DBMS 기본정보( 종류, 버전, 호스트 이름, 몇개의 서버를 사용하는지
- 리플리케이션, 클러스터, 프락시 등을 사용하는지
- 그 외 접속정보와 애플리케이션에서 접속하는 방법, 백업 정책 등도 기술한다.

#### Object Relation Mapping

- 비즈니스 규칙이 어떤 방법으로 구현되었는지
- 데이터 유효성 검증, 데이터 변환, 로깅, 캐싱 , 프로파일링을 어떻게 하는지 기술한다.

#### 소스 코드 제어

- 개발 중 데이터베이스는 설계는 지속적으로 변경되므로
- VCS를 사용해 DB 코드에 대해 변경 내역을 저장하여 추적해야 한다.
- 기존 배포가 완전히 망가지더라도 프로젝트를 다시 빌드하고 배포할 수 있도록 필요한 모든 것을 준비해야 한다.

**DDL**

#### 입증 책임 : 테스트

데이터베이스의 구조와 동작을 애플리케이션 코드와 독립적으로 검증한다.

#### 체크리스트

- 테이블, 컬럼, 뷰

  - 테이블과 뷰가 실제로 존재하는지
  - 테이블, 뷰, 컬럼을 추가할때마다 객체가 있는지를 확인하는 테스트를 추가한다.(잘 추가 되었는지 확인하는 테스트를 의미하는 듯 하다.)

- 제약조건
  제약조건이 제대로 동작하는지 확인하기 위해 그것을 위반하는 코드를 통해 테스트한다.

- 트리거
  트리거를 실행하는 문장을 통해 시나리오를 테스트한다.

- 쿼리
  쿼리 결과가 기대한 컬럼과 데이터가 있는지 테스트하는 코드를 추가한다.

### 패턴 25 마법의 콩

액티브 레코드 패턴은 쿼리가 단순한 경우 편리하지만
애플리케이션에서 해결해야 하는 모든 문제에는 적합하지 않다.
