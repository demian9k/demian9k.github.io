친절한 SQL 튜닝 정리 - 4

친절한 SQL 튜닝 정리 - 4

이 문서는 조시형의 저서 친절한 SQL 튜닝을 개인적으로 참조하기 위해 요약,정리한 것으로 분산된 내용이 많아 책 없이 읽을 수 없는 문서입니다.



4장 조인 튜닝

4.1 NL 조인

NL조인은 Nested Loop 의 줄임말이다. 말 그대로 두 테이블을 두개의 중첩된 반복문으로 검색하고, 기준키(FK)가 일치하는 레코드를 찾는 것이다.
두 테이블은 FK를 통해 서로의 테이블을 참조할 때 인덱스 스캔과 TFS를 사용할 수 있지만 일반적으로 인덱스 스캔을 사용한다. Table Full Scan은 NL 방식에서 너무 비효율적이기 때문이다.

NL 조인에서 두 테이블의 방향에 따라 테이블의 명칭이 있다. 중첩된 반복문에서 바깥쪽에 있는 테이블을 outer 라고 하고, 안쪽에 있는 테이블을 Inner 라고 한다.
outer는 한번만 스캔되지만, inner는 outer 레코드 갯수만큼 스캔되는 것이다.

책에서는 아래와 같은 SQL을 보여주었는데

select * from
table1, table2
where table2.fk = table1.fk

table1이 outer( driving ) 이고 오른쪽에 있는 테이블이 inner 이다. (실행계획에서 그렇게 나타나며, 조건절에서 table2가 fk라도 from에서 선언된 순서가 먼저인것 같다.)

(sybase 매뉴얼에서 --> 실행계획이 없이 결론지은 판단이다.) left join문에서는 왼쪽이 outer 이고, right join에서는 오른쪽이 outer 이다.

T1 left join T2
T2 right join T1

다시 인덱스로 NL을 한다는 것은 outer의 레코드 하나마다 인덱스 검색을 하고, inner 레코드 하나마다 인덱스를 사용한다는 것이다.
NL은 다수의 Index range scan을 사용하므로 인덱스 성능에 크게 영향을 받는다.

힌트를 통해 조인 순서를 바꾸거나 조인방법을 바꿀 수도 있다.

NL조인 튜닝포인트

  1. outer index 읽은 후 outer table을 접근할 때의 랜덤액세스 정도 측정
    여기에서 랜덤액세스가 많이 발생하는데 특정 컬럼으로 필터링 되는 비율이 높다면 특정 컬럼을 인덱스에 추가하여 랜덤액세스를 줄이는 방안을 고려한다.

  2. inner index 탐색(조인 액세스 횟수)
    inner index를 탐색하는 횟수는 outer 테이블을 읽고 필터링한 결과건수에 의해 결정된다.

  3. inner index를 읽고 나서 inner table을 액세스 하는 부분
    1번의 경우와 마찬가지로 특정 컬럼에 의해 필터링 되는 비율이 높다면 특정 컬럼을 인덱스에 추가하여 랜덤액세스를 줄이는 방안을 고려한다.

  4. outer index에서 얻은 결과 건수에 의해 전체 일량이 좌우된다.
    outer index에서 얻은 결과 건수에 따라 outer table 랜덤액세스 횟수(1번), inner index 탐색횟수(2번항목) inner 랜덤액세스 횟수(3번), inner index 탐색횟수(2번항목)

소트 머지 조인

PGA는 오라클 서버 프로세스(사용자 SQL 요청을 처리)에 할당된 공유하지 않는 메모리 영역이며 프로세스에 종속적인 고유 데이터를 저장하는 용도로 사용한다. 독립적인 공간이므로 래치 메커니즘이 불필요하고 SGA 버퍼캐시에서 읽을 때보다 빠르다.

PGA의 공간이 작을때는 Temp TableSpace 를 이용한다.

테이블 2개를 조인할때 소트 머지 조인은 2 단계를 거친다.

  1. 소트 단계
    원본 SQL의 조인 문장에서 각자 테이블별로 데이터를 읽고 조인컬럼별로 정렬한다음 PGA의 Sort Area에 저장한다. (이는 실시간으로 인덱스를 만드는 것과 비슷하다)

  2. 머지 단계
    각 테이블별로 저장된 Sort Area의 데이터를 스캔하면서 조인한다. 조인 단계에서 중요한 것은 NL과 다르게 데이터가 정렬되어 있으므로 조인에 실패하는 레코드를 만나면 멈추고 다음 ROW 조인을 진행한다. 소트 머지 조인이 대량 데이터 조인시 더 빠른 이유는 소트단계에서 생성한 데이터들은 일반적인 Index를 사용한 NL 조인의 절차인 래치 획득 및 버퍼캐시 체인 스캔 과정을 거치지 않아서 그렇다.
    소트 머지 조인도 조인 대상 집합을 읽을 때는 DB 버퍼캐시를 경유하고, 이때 인덱스를 이용하기도 하므로 이때 생기는 버퍼캐시 탐색비용과 랜덤액세스 부하도 피할 수 없다.

소트 머지 조인을 사용해야 하는 경우

조인 조건식이 등치 조건이 아닐 때의 대량 데이터 조인 조인 조건식이 아예 없는 조인( Cartesian Product )

해시 조인

해시 조인 2단계

  1. Build

    • 작은 쪽 테이블(Build Input)을 읽어 해시 테이블을 생성한다.

  2. Probe

    • 큰 쪽 테이블(Probe Input)을 읽어 Build 단계에서 만들어진 해시 테이블을 탐색하면서 조인한다.

빌드 단계에서 조인 컬럼이 해시 테이블의 키가 되며 해시 테이블은 PGA의 Hash Area에 저장한다. (해시 테이블에는 키와 데이터(실제 컬럼) 이 모두 저장된다.)

소트 머지 조인과 동일하게 대량 데이터 조인시 NL조인보다 빠르지만 차이점은 소트 머지 조인이 테이블을 전부 PGA에 정렬해서 담는것에 비교해서 해시 조인에서는 테이블 중 하나(작은 것 Build Input)만 PGA에 담으므로 상대적으로 PGA에서 부담하는 공간의 양이 적다.

조인 메서드 선택 기준

소량일 때는 NL조인을 선택한다.

대량일때는 소트 머지 조인이나 해시 조인 중 하나를 선택하는데 조인 조건식에 따라 등치이면 해시 조인을, 아니면 소트 머지 조인을 선택한다.
대량 데이터라는 것은 최적화된 NL 조인을 했을 때 랜덤 액세스가 많아 만족할만한 성능을 낼 수 없다면 대량 데이터처리로 판단한다.

수행빈도가 매우 높은 쿼리에에서의 선택기준에서는 최적화된 NL조인이 해시 조인과 성능이 비슷하면 NL조인을 선택하고 그렇지 않다면 해시 조인을 선택한다.

NL조인이 우선이 되야 하는 이유는 인덱스는 서버 프로세스들이 영구적으로 공유하는 자료구조인데 비해 해시 테이블은 한번 생성하고 조인이 끝나면 소멸하기 때문이다.
그래서 수행빈도가 높은 쿼리는 CPU가 메모리 사용률이 크게 증가하고, 해시 테이블을 만드는 과정에 래치 경합또한 발생한다.
해시 조인을 사용해야 할 때는 수행빈도가 낮고, 대량 데이터 접근으로 쿼리 수행시간이 오래 걸리는 경우에 사용해야 한다.

서브쿼리 조인

사용자가 작성한 SQL은 옵티마이저에 의해 최적화에 유리한 형태로 변환된다.(SQL Transformation)
서브쿼리는 3가지가 있다.

  • 스칼라 서브쿼리
    • 값을 하나만 반환하는 서브쿼리, SELECT-LIST 절에서 사용하지만 컬럼이 올수 있는 대부분의 위치에 사용할 수 있다.
  • 인라인 뷰
    • FROM 절에서 사용한 서브쿼리
  • 중첩된 서브쿼리
    • 결과집합을 한정하기 위해 WHERE 절에 사용된 서브쿼리

하나의 SQL 문장은 하나의 쿼리 블록이며 SQL 문장을 구성하는 서브 쿼리 또한 하나의 쿼리 블록이다.
하나의 문장은 하나의 메인 쿼리 블록과 하위 쿼리 블록들로 이루어진다고 볼수 있다.
옵티마이저는 쿼리 블록 단위로 최적화를 수행한다.

Filter Operation

메인쿼리와 서브쿼리 사이에는 부모 자식이라는 종속적이고 계층적인 관계가 존재한다.
서브쿼리는 메인쿼리에 종속되므로 단독으로 사용할 수 없으며 메인쿼리 건수만큼 값을 받아 반복적으로 필터링하는 방식으로 실행된다.

이때 서브쿼리가 필터링하는 것을 Filter Operation이라고 부르며, 이것은 NL과 동일하게 중첩 반복문이라는 점은 같지만 차이가 존재한다.

  1. 메인쿼리의 로우가 서브쿼리의 로우에 조인하는 순간 멈추고 메인 쿼리의 다음 로우를 진행한다
  2. 필터는 캐싱기능이 있다. 서브쿼리 조인 에 따른 반환값(true, false)이 캐싱되어 서브쿼리를 수행하지 않을 수 있다.

Subquery Unnesting

Unnesting은 서브쿼리가 포함된 메인 쿼리블록을 SQL Transformation 하여 서브쿼리가 메인 쿼리블록에 병합하는 것을 말한다.
SQL Transformation 없이 SQL을 처리하는 것을 nesting 이라고 부른다
일반적으로 옵티마이저는 최적화에 유리한 unnesting을 사용한다.

unnesting된 서브쿼리를 메인 쿼리 집합보다 먼저 처리할 수 있다.
서브쿼리를 unnesting 해서 메인쿼리와 같은 수준으로 만들면(병합하면) 다양한 조인 메소드를 선택하거나 조인 순서를 정할 수 있다.)
조인 순서와 메서드를 정할 수 있다는 것은 옵티마이저에게 더 좋은 선택지를 만들어 주는 효과를 가진다.

Subquery pushing

필터 방식으로 처리하는 경우에 적용할 수 있는 방법이다.
서브쿼리가 실행계획상에서 나중에 처리됨으로 인해 조인 과정에서 비효율적인 I/O가 발생하는 경우에 서브쿼리를 먼저 실행시키는 것을 pushing 이라고 한다.

pushing은 서브쿼리 안에 PUSH_SUBQ 힌트로 부여함으로서 가능하며, Filter Operation을 사용하므로 NO_UNNEST 힌트 또한 같이 기술하는 것이 올바른 사용법이다.

Join Predicate Pushdown

서브쿼리를 사용할 때 메인쿼리 블록과 서브쿼리 블록을 NL 조인 수행시 메인 쿼리의 결과를 서브쿼리 안으로 밀어넣는 것

책에서는 이 예제를 설명하기 위해 세 가지 예제를 보여주었다.

  1. A Table 과 B Table이 있으며, 쿼리는 A테이블을 첫번째 조인 대상으로, 두번째 테이블은 B 테이블의 group by절이 사용된 인라인 뷰(서브 쿼리 블록)로 생성하였다.
    그런데 메인 쿼리 블록 조건절에 인라인 뷰와의 조인 횟수를 줄일 수 있는 조건절이 있다.
    인라인 뷰에서는 메인 쿼리 블록의 데이터를 접근할 수 없어 인라인 뷰에서는 해당 조건을 사용하지 못하고 조인을 시도하게 되어 비효율이 발생한다.

이 문제를 해결하기 위해 첫번째 시도한 방법으로 서브쿼리 안에 merge 힌트를 사용하였다.
merge는 서브쿼리를 일반적인 조인형태로 SQL 변환하는 것을 말한다.

책에서는 merge 힌트로 변환된 쿼리에 인덱스가 최적으로 생성되어 있다고 가정했을 때, 인라인 뷰에서 메인 쿼리 블록에 접근하지 못해 비효율적 조인을 하게 되는 것을 피하게 되지만 merge로 인해 group by 절이 바깥으로 나와있어서 조인에 성공한 전체 집합을 group by 해야 데이터를 출력할 수 있다.
이 점은 부분범위(Partial Range) 처리가 불가능하다.
