aws-fundamental-1

region

AWS 에서 제공하는 서비스는 전지구적으로 제공되고 아래와 같은 이유로 region 이라는 각 지역을 논리적 개념으로 나누었다.
물리적으로는 데이터센터들의 그룹이다.

  • 고객과의 근접성: 네트워크 지연을 줄이고 AWS 리소스의 성능과 응답성을 향상시키기 위해서
  • 규정 준수 및 데이터 주권: 국가 및 지역마다 정보보안에 대한 법을 지키기 위해 현지에 데이터센터를 두는 것
  • 재해복구(Disaster Recovery) 및 고가용성(HA) : 여러 리전에 데이터와 애플리케이션을 복제하여 구현한다.
  • scaling and elasticity: 리전을 사용해 워크로드를 분산하여 수평으로 확장할 수 있다.

주로 국가에 1~2개 설치되나 땅이 넓거나 사용량이 많은 경우 더 많은 region이 있다.

region 이름은 아래와 같은 형태이다.

  • us-east-1
  • ap-northeast-1(도쿄)
  • ap-northeast-2(서울)

AZ

Available Zone 가용영역

AZ는 동일한 리전 내 다른 AZ의 장애로부터 격리되도록 설계된 리전 내의 고유한 데이터 센터들이다.
각 AZ에는 자체 전원, 네트워킹 및 다른 AZ에 대한 연결(connectivity to other AZs)이 있어 AWS에서 실행되는 애플리케이션의 고가용성(HA)과 내결함성(fault tolerance)을 보장한다.

이미지 출처: aws docs

위 그림은 각 계정에 표시되는 us-east-1a 같은 az 이름이 실제 region과 각자 다르게 매핑된다는 것을 보여준다.
11111 에서는 us-east-1a 가 use1-az1에 연결되었지만 22222에서는 us-east-1c 라는 이름으로 연결되었다.

IAM

Identity and Access Management (사용자 계정 그룹 및 정책 관리)

AWS 계정은 계약에 관한 내용이고 IAM는 권한별 접근 자격이다.

세가지 종류가 있다.

IAM User

일반적으로 최소한의 기능을 부여하고 필요한 사람에게만 전달하여 운영한다. 불필요한 권한을 부여하거나 하나의 IAM을 공유해서 사용하는 것도 바람직하지 않다. audit에서 누가 어떤 행위를 했는지 검사할 때 사용되기 때문이다.

IAM Group

그룹을 사용하면 같은 권한을 부여하고 싶은 사용자를 일괄로 관리할 수 있다.

IAM Policy

어떤 AWS 리소스(EC2 인스턴스, S3의 디렉토리) 에 대해서 어떤 operation 등을 허가할지 말지를 설정하는 것이다.
실행자(User, Role, Group)가 어떤 일을 할 수 있는가? 의 형태(자격 기반 형태) 로 설정이 가능하고
operation 대상(서버 및 폴더) 에 대해서 무엇을 허가할 것인지의 형태(리소스 기반 정책)로도 가능하다.
S3에서는 읽기, 추가하기, 삭제하기 등 세밀하게 설정하거나 특정한 IP에서만 허가할 수도 있다. IAM정책은 너무 자세하게 설정이 가능해서 그것 자체가 일이 되기 때문에 템플릿 형태인 AWS 관리 정책을 사용할 수 있다.

Cloud Watch

리소스 상태 감시

리소스의 상태 예)
CPUUtilization, Disk Read/Write Ops, Disk Read/Write Bytes, NetworkPackets In/Out

어떤 상태값이 임계를 초과하는 경우 어떤 동작을 하게 만들 수 있다.

  • email 송신,
  • ec2 인스턴스 시작, 정지
  • auto scaling
  • aws Lambda 함수 실행

Cloud Watch Logs 를 통해 각종 로그를 기록할 수 있다.
EC2 인스턴스에 에이전트를 설치하여 인스턴스내 로그를 기록할 수 있다.
CloudTrail 이라는 aduit 서비스를 enable하면 누가 어떤 리소스에 접근했는지 기록으로 남길 수 있다.


EC2

Elastic Compute Cloud 범용 서버

매니지드 서비스가 아니므로 OS 및 소프트웨어는 직접 설치하고 운영해야 한다.

EC2는 VM으로서 Region-AZ에 설치될 수 있으므로 어디서든 접속할 수 있고 물리서버에 비해 백업이 매우 쉽다.

AMI(Amazon Machine Image)를 통해 인스턴스를 생성한다. 인스턴스에 접속할때는 key pair를 필요하다.

EC2는 스토리지로 여러가지를 선택할 수 있다.

  • instance store

    • 임시용도의 블록 스토리지로서 EC2가 재부팅될때는 데이터가 유지된다. 그러나 중지, 종료, 최대절전모드, 드라이브 오류시 데이터가 손실된다.

  • EBS

    • 인스턴스의 중지나 종료에 상관없이 데이터가 유지된다.
    • 인스턴스와 독립적이므로 EBS 볼륨을 Detach 하여 다른 인스턴스에도 사용할 수 있다.(루트인 경우 Detach 불가)

  • EFS

  • S3

또한 Security Group(가상 방화벽)을 EC2에 적용하여 프로토콜, 포트, IP 주소 등으로 네트워크의 접근제어가 가능하다.


네트워크

EC2 인스턴스를 배치할 네트워크로 AWS의 VPC에서 선택해야 한다.

인스턴스에는 private IP가 부여되고 NAT에 의해 public ip로 변환된다. (리전별로 하나씩 존재하는 기본 VPC에는 이미 NAT가 설정되어 있다.) 인스턴스에 public ip를 부여하려면 Elastic IP라는 리소스를 사용하여 할당할 수 있다.

VPC에서 서브넷을 선택하면 AZ와 private IP 범위가 결정된다.


Storage 유형


Block

  • 데이터를 로지컬 볼륨의 fixed size block에 저장한다.

  • 다양한 path를 갖고 있어 보통의 스토리지보다 성농이 높고 지연이 낮다.

  • 파티션을 사용할 수 있어 OS 호환성이 좋다.

  • SAN, VM이나 DB같이 대규모로 사용하기에 적합하다.

    단점

    • 비용이 높다.
    • 메타데이터 처리가 제한적이라서 데이터수준이 아니라 app, DB 수준에서 작업이 필요해 관리 부담이 있다.

File

일반적인 파일 저장 방식, 계층적인 디렉토리 및 파일 구조로 데이터 양이 증가할 수록 성능이 저하된다


Object

  • 데이터를 오브젝트 형태로 다룬다.
  • 오브젝트 단위는 flat 주소 공간에 uid를 통해 접근할 수 있다. 그래서 빠르고 scalability 가 좋다.
  • file, block 에 비해 다양한 메타데이터 추가가 가능하여 검색이 용이하다.
  • 큰 볼륨의 비정형 데이터(멀티미디어, 백업, 아카이브 등) 를 저장하는 데 일반적으로 사용된다.
  • 오브젝트는 수정이 되지 않고 append하는 방법을 사용하므로 자주 변경되는 데이터에 적합하지 않고 한번 저장하고 변경이 적은 데이터에 적합하다.

EBS

Elastic block storage
EC2 인스턴스에 attach 해서 쓸수 있는 영구적인 유형의 스토리지이다.

FeatureDescription
Elastic volumes볼륨 크기를 간단하게 조정한다.
Snapshot어떤 시점의 데이터를 통째로 저장하는 기능이다.
Data lifecycle manager일정별로 스냅샷을 생성,삭제
Optimized instances특정 인스턴스 유형을 최적화하여 읽기 쓰기를 고속화
Encryption데이터 볼륨, 부팅 볼륨 및 스냅샷을 암호화한다.

RDS

relational database service

Managed 관계형 DB 서비스

mariadb, Amazon Aurora, oracle, MS SQL Server, PostgreSQL 등의 다양한 종류를 지원한다.

Elastic IP

AWS가 제공하는 공인 IPv4 주소이다.

EC2 인스턴스는 정지 후 재시작하면 기본 VPC의 NAT 때문에 공인 IP 주소가 변경되므로 서버로서 사용하려면 고정 IP가 필요할때 사용한다.

ElasticIP 는 AWS 계정에 연결되어 있고 이것을 instance에 할당해서 사용한다. 인스턴스를 삭제해도 할당이 해제될 뿐 AWS 계정의 IP는 삭제되지 않는다

ELB

Elastic Load Balancer

네트워크 부하 분산기이다.

3가지 종류가 있다.

ALB(Application LB)

http, https 프로토콜을 지원하는 로드밸런서이다. OSI model의 7계층에서 동작한다.

기존의 L7 스위치와 비슷하다.

NLB(Network LB)

OSI model의 transfort 계층에서 동작하여 TCP, TLS 프로토콜을 지원한다/ 패킷까지만 inspect하기 때문에 ALB보다는 기능이 제한된다. 분배 대상의 static ip 주소를 설정할수 있고 서버에 접속한 클라이언트의 IP 주소를 그대로 서버에 전송하도록 할 수 있다.

기존의 L4 스위치와 비슷하다.

CLB(Classic LB)

AWS에서 ELB이전에 사용하던 LB이다. 지원하는 프로토콜이 많지만 잘 쓰이지 않는 LB 이다. TCP SSL/TLS HTTP, HTTPS 프로토콜을 지원한다.

snapshot

서버 디스크의 한 순간의 데이터를 모두 보존한 파일이다. 통째로 보존하므로 데이터, 소프트웨어, OS, config 등도 모두 포함된다.
OS 변경 등을 할때 바로 rollback을 하기 위해 backup 용도로 사용하거나 AWS 에서는 CUSTOM AMI를 만들기 위해 사용하기도 한다.
EBS 볼륨의 데이터를 스냅샷으로 보존할 수 있는데 incremental snapshot approach를 사용하여 비용을 줄인다.

스냅샷을 삭제하면 해당 스냅샷의 고유한 데이터만 삭제되고 최초 데이터를 삭제할 경우 두번째 스냅샷과 비교하여 변경되지 않는 부분만 두번째 스냅샷과 합쳐진 후 삭제된다.

Auto scaling

어떤 리소스의 접근 상태에 따라 인스턴스 갯수를 늘리거나 줄이는 기능 일반적으로 EC2 인스턴스에 많이 사용되나 아래 리소스들에도 적용할 수 있다.

  • ECS task( 매니지드 컨테이너 오케스트레이션 서비스로 EC2 클러스터를 기반으로 한다. )
  • DynamoDB Tables
  • Aurora replica
  • Spot fleets

리소스의 상태는 CloudWatch의 메트릭에서도 참조하여 동작할 수 있다.

일반적으로 EC2의 auto scaling 하려면 설정해줘야 하는 것은 아래와 같다.

  1. auto scaling group 생성
  2. group에 인스턴스 최소와 최대 갯수 설정
  3. AMI, keypair, security gorup 설정

scaling 동작에는 3가지가 있다.

  1. ec2 인스턴스가 정지한 경우 분리하고 새로운 ec2 인스턴스 생성
  2. 일정에 맞춰 스케일링
  3. cpu와 네트워크의 부하를 참고해 지정한 임계값을 넘을 때 자동으로 scaling
