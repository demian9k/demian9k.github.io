K8S manifest - ServiceAccount

UserAccount는 GKE의 구글 계정, EKS의 IAM 등과 연결된 리소스로 쿠버네티스의 관리 대상이 아니다.

또한 UserAccount는 cluster-evel resource로 namespace의 영향을 받지 않는다. ( container->Pod->Node->Cluster )

ServiceAccount는 k8s 에서만 사용되는 것으로 파드에서 실행되는 프로세스를 위해 할당된다. 네임스페이스와 연결된 리소소이며, 파드가 1개 실행되면 반드시 SerivceAccount 1개를 할당해야 하며, 이 ServiceAccount를 기반으로 Authentication/Authorization를 수행한다. 지정하지 않는 경우 default Service Account가 할당된다.

명령어로 생성

kubectl create serviceaccount sample-serviceaccount

imagePullSecrets 필드는 인증이 필요한 private image registry에서 이미지를 가져오기 위해 Secret을 설정한다.

apiVersion: v1
kind: ServiceAccount
metadata:
  name: sample-serviceaccount
  namespace: default
imagePullSecrets:
  - name: myregistrykey

생성된 sample-serviceaccount를 확인하면 기본저긍로 생성되는 Secret이 생성되어 있다. kubernetes.io/service-account-token 타입의 Secret이다.

apiVersion: v1
kind: ServiceAccount
metadata:
  name: sample-serviceaccount
  namespace: default
imagePullSecrets:
  - name: myregistrykey
secrets:
  - name: sample-serviceaccount-token-b76nf

이 토큰은 명시적으로 지정할 수도 있고, 변경하려면 Secret을 삭제하면 재생성되는 특징이 있다. 이 secret은 token과 ca.crt(인증서)로 구성되어 있다.

kubectl get secrets sample-serviceaccount-token-b76nf -o yaml

Service Account는 이 token으로 kubernetes API에 대한 인증 정보로 사용할 수 있다. 따라서 Pod에 할당된 ServiceAccount의 할당된 권한이 그대로 파드에 할당된다.

Pod에 ServiceAccount를 명시적으로 지정하기 Pod를 apply 하면 token이 볼륨으로 자동으로 포함된다.

apiVersion: v1
kind: Pod
metadata:
  name: sample-serviceaccount-pod
  namespace: default
spec:
  serviceAccountName: sample-serviceaccount
  containers:
    - name: nginx-container
      image: nginx:1.16

볼륨으로 mount 되므로 mountPath에는 토큰 및 인증서가 파일로 존재한다.

kubectl exec -it sample-serviceaccount-pod -- ls /var/run/secrets/kubernetes.io/serviceaccount/

만약 ServiceAccount에 RBAC 설정이 되어 있고, container에서 다음과 같이 실행하면 API 접근을 위한 Authentication/Authorization이 성공하고, default namespace로 동작하는 파드를 볼 수 있다.

kubectl exec -it sample-serviceaccount-pod -- bash

apt update && apt -y install curl

TOKEN=`cat /var/run/secrets/kubernetes.io/serviceaccount/token`

curl -H "Authorization: Bearer ${TOKEN}" \
--cacert /var/run/secrets/kubernetes.io/serviceaccount/ca.crt \
https://kubernetes/api/vi/namespaces/default/pods

token automount

토큰은 자동으로 Mount되지만 automount 를 off 할 수도 있다.

apiVersion: v1
kind: ServiceAccount
metadata:
  name: sample-serviceaccount-noautomount
  namespace: default
automountServiceAccountToken: false #토큰은 automount 되지 않는다.
---
#그래서 token을 Pod에서 mount 하려면 아래처럼 automountServiceAccountToken를 true로 설정한다.
# ServiceAccount와 Pod 모두 automount 설정이 가능하지만 둘 다 설정하면 Pod 설정이 우선시된다.
# ServiceAccount에서 false로 설정하면 white list 방식, true로 설정하면 blacklist 방식으로 운영된다.
apiVersion: v1
kind: Pod
metadata:
  name: sample-serviceaccount-noautomount-pod
  namespace: default
spec:
  serviceAccountName: sample-serviceaccount-noautomount
  automountServiceAccountToken: true
  containers:
    - name: nginx-container
      image: nginx:1.16

쿠버네티스의 ServiceAccount 및 토큰 자동 마운팅과 관련하여, "WhiteList"와 "BlackList"라는 용어는 Pod에 Mount되는 Secret을 제어하기 위한 다양한 접근 방식을 의미한다.

  1. WhiteList:
  • ServiceAccount에서 automountServiceAccountToken이 false로 설정되면, WhiteList 모드에서 동작한다.
  • 이 모드에서는 기본적으로 Pod에 Secret이 자동으로 Mount되지 않는다.
  • 특정 Secret을 Pod에 Mount하려면, Pod의 구성에서 imagePullSecrets 또는 secrets 필드를 사용하여 명시적으로 정의해야 한다.
  • Pod의 구성에 명시적으로 나열된 Secret만 Pod 내부에서 사용할 수 있다.
  1. BlackList:
  • ServiceAccount에서 automountServiceAccountToken이 true로 설정되면, BlackList 모드에서 동작한다.
  • 이 모드에서는 ServiceAccount와 관련된 모든 Secret이 기본적으로 Pod에 자동으로 Mount된다.
  • 특정 Secret을 Mount에서 제외하려면, Pod의 구성에서 false 값으로 automountServiceAccountToken 필드를 사용하여 해당 Secret을 명시적으로 정의해야 한다.
  • Pod의 구성에서 명시적으로 제외되지 않은 모든 Secret은 자동으로 Pod에 Mount된다.

요약하면, WhiteList 모드는 Mount하려는 Secret을 지정해야 하는 반면, BlackList 모드는 Pod의 구성에서 명시적으로 제외되지 않는 한 ServiceAccount와 관련된 모든 Secret을 자동으로 Mount한다.

Client Library에서의 인증

client-go 라이브러리는 k8s클러스터에 인증할 때 2가지 방식을 제공한다.

In-Cluster Config 이 방법은 동작하는 Pod a가 자신의 k8s 클러스터 c에 대해 a Pod에 할당된 ServiceAccount Token을 사용하여 c클러스터의 kubernetes API와 통신하기 때문에 명시적으로 인증 정보를 지정할 필요가 없다. 또한 파드에 할당된 ServiceAccount에서 Authentication/Authorization 을 위하여 파드 생성 환경에 따라 동적으로 권한을 변경할 수 있다.

다음 코드는 go에서 incluster config를 사용하는 방법이다.

config, err := rest.InClusterConfig()
clientset, err := kubernetes.NewForConfig(config)

out-of-cluster configuration 사용하기 명시적으로 인증 정보를 지정하는 방법으로 클러스터 외부에서 kubernetes API와 통신하려면 이 방법을 사용한다.

config, err := clientcmd.BuildConfigFromFlags("", *kubeconfig)
clientset, err := kubernetes.NewForConfig(config)

docker registry 인증 자동 설정

imagePullSecrets 가 지정된 ServiceAccount를 할당한 Pod가 동작하면, 자동으로 Pod의 imagePullSecrets로 사용된다.

# imagePullSecret을 가진 Service Account
# imagePullSecrets는 리스트이므로 여러개 할당도 가능하다.
apiVersion: v1
kind: ServiceAccount
metadata:
  name: sample-serviceaccount-pullsecret
imagePullSecrets:
  - name: sample-registry-auth
---
#Pod에 ServiceAccount 설정
apiVersion: v1
kind: Pod
metadata:
  name: sample-serviceaccount-pullsecret-pod
spec:
  serviceAccountName: sample-serviceaccount-pullsecret
  containers:
    - name: nginx-container
      image: nginx:1.12
