K8S manifest - PersistentVolume

PersistentVolume

일반 Volume은 파드 정의 안에 직접 지정되지만 PV는 K8S manifest로 생성한 후 사용한다.

PV 또한 플러그인 형태로 여러가지 플러그인을 지원한다.

  • GCE Persistent Disk
  • AWS EBS
  • Azure File
  • nfs
  • iSCSI
  • Ceph(RBD, CephFS)
  • OpenStack Cinder
  • GlusterFS
  • Container Storage Interface(CSI)
  • 최초 Storage를 연결하려면 provider용 플러그인이 변경될 때 kubernetes의 코드 자체도 변경해야 하는 문제가 있었고 이것을 인터페이스화 하여 그 문제를 해소하기 위해 개발되었다.
  • 그래서 GCP에는 GCE Persistent Disk Plugin(기존 플러그인) 과 CSI 인터페이스를 구현한 Persistent Disk CSI Driver 2가지가 존재한다.

아래 PV는 gce Persistent Disk를 사용한 example manifest이다. 먼저 GCP에서 Persistent Disk가 있는 상태에서 apply 해야 한다.

apiVersion: v1
kind: PersistentVolume
metadata:
  name: sample-pv
  labels: # 레이블
    type: gce-pv
    environment: stg
spec:
  capacity: # 용량
    storage: 10Gi
  accessModes: # 접근 모드
    - ReadWriteOnce
  persistentVolumeReclaimPolicy: Retain # Reclaim Policy
  storageClassName: manual # 스토리지클래스
  # PersistentVolume 플러그인별 설정(GCE 영구 디스크 예)
  gcePersistentDisk:
    pdName: sample-gce-pv
    fsType: ext4

Label

동적 프로비저닝을 사용하지 않고 PV를 생성하면, PV종류를 알수 없게 되기 때문에 type/environment/speed 등의 레이블을 사용하는 것이 권장된다. 레이블을 지정하지 않으면 PV에서 할당할 때 사용자가 원하는 볼륨을 지정하기가 어렵다.

capacity

capa를 지정할 때 주의할 점은 동적 프로비저닝을 사용할 수 없는 환경에서는 작은 용량의 PV도 준비해 두어야 한다는 것이다. 작은 PV가 없으면, 그와 가장 비슷한 PV가 할당되기 때문이다.

accessMode

  • ReadWriteOnce(RWO)

  • 단일 node 에서만 read/write 를 허용한다.

  • ReadOnlyMany(ROX)

  • 여러 노드에서 Read 가능

  • 하나라도 쓰기 요청이 있는 파드가 있다면 다른 노드에서 마운트가 불가능하므로 pvc를 지정할 때 readOnly로 지정한다.(spec.volumes[].persistentVolumeCliam.readOnly)

  • ReadWriteMany(RWX)

  • 여러 노드에서 Read/Write 가능

ReadOnlyMany로 영구 볼륨을 사용하는 예제 PVC를 먼저 적용한 후 deployement를 적용한다.

kind: PersistentVolumeClaim
apiVersion: v1
metadata:
  name: sample-pvc
spec:
  selector:
    matchLabels:
      type: gce-pv
    matchExpressions:
      - { key: environment, operator: In, values: [stg] }
  resources:
    requests:
      storage: 4Gi
  accessModes:
    - ReadWriteOnce
  storageClassName: manual

---
apiVersion: apps/v1
kind: Deployment
metadata:
  name: sample-pv-readonlymany
spec:
  replicas: 3
  selector:
    matchLabels:
      app: sample-app
  template:
    metadata:
      labels:
        app: sample-app
    spec:
      containers:
        - name: nginx-container
          image: nginx:1.12
          ports:
            - containerPort: 80
              name: 'http'
          volumeMounts:
            - mountPath: '/usr/share/nginx/html'
              name: nginx-pvc
              # readOnly: true
      volumes:
        - name: nginx-pvc
          persistentVolumeClaim:
            claimName: sample-pvc
            readOnly: true

Reclaim Policy

PV를 사용한 후 처리 방법(삭제, 재사용 등)을 제어하는 정책이다. (PVC에서 사용된 후 PVC가 삭제되었을 때 PV의 동작을 설정한다.)

Delete

  • PV 가 삭제된다.(데이터 삭제)
  • GCP/AWS/OpenStack 등에서 확보되는 외부 볼륨의 동적 프로비저닝 때 사용되는 경우가 많다.

Retain

  • PV는 삭제되지 않고 유지된다.
  • 다른 PVC에 의해 PV가 다시 mount 되지 않는다.

Recycle

  • PV 데이터를 삭제하고 재사용 가능한 상태로 만든다.
  • 다른 PVC 에서 다시 마운트 가능하다.
  • 이 정책은 k8s에서 deprecated 되어 사용되지 않는다(동적 프로비저닝이 권장된다.)

Delete Policy는 PVC에 의해 매칭되면 Bound 상태가 되며 PVC가 삭제되면 Released 상태가 되며 PV가 삭제되며, 다른 PVC와 조건이 맞더라도 사용이 불가능해진다.

Retain은 Bound 되었던 PVC가 삭제되더라도 Released 상태로 전이할 뿐 PV와 그 데이터는 삭제되지 않는다. Released 상태가 된 PV는 PVC와 조건이 맞아도 사용이 불가능해지며 다시 Bound 시키려면 새 PV를 생성해야 한다.

StorageClass

GKE는 기본으로 kubernetes.io/gce-pd 라는 provisioner 가 설정된 스토리지클래스가 정의되어 있다. provisioner는 cloud 환경의 api를 호출하는 등 동적으로 PV를 프로비저닝하는 도구로 GKE에서는 기본 설정 그대로 PVC를 사용하면 자동으로 동적 프로비저닝이 된다.

자동으로 동적 프로비저닝을 차단하려면 아래의 manifest를 적용하면 된다.

apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: manual
provisioner: kubernetes.io/no-provisioner

mountOption

영구 볼륨에 종류에 따라 spec.mountOptions를 설정할 수 있다.
