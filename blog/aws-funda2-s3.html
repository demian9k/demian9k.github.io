S3 용어

termDescription
객체S3의 엔터티 단위이다. 쉽게 말하면 텍스트나 이미지 등의 파일을 말한다.
버킷객체를 저장하는 컨테이너이다. 모든 객체는 버킷이 저장된다.
버킷명S3버킷의 명칭은 다른 AWS 사용자를 포함하여 유일한 이름이어야 한다. 웹 서버로 사용할 경우는 도메인명이 버킷명이 된다.
객체 키객체 식별자이다. 모든 객체는 반드시 한 개의 키를 가진다. 버켓, 객체, 키, 버전을 조합하여 객체를 고유하게 식별한다.
객체 메타데이터이름과 값의 세트이다. 객체를 업로드할 때 설정할 수 있다. 객체 키는 S3에서 사용하는 file id 지만 메타데이터는 사람이 파일을 쉽게 관리하기 위한 데이터이다.
리전버킷의 물리적인 보관 지역이다.
S3 데이터 일관성 모델S3는 가용성을 유지하기 위해 데이터를 자동 복제하여 저장한다. 쓰기 지연에 의한 데이터 불일치가 발생하지만 데이터 일관성 모델이 이것을 방지한다. 복제가 모두 반영되기에는 시간이 걸릴 수 있다.
버전 관리데이터의 여러가지 버전을 보관한다. 다른 버전을 객체로 취급하는 것이 가능하다.
로그버킷 단위나 객체단위의 로그를 기록할 수 있다. 객체 수준의 경우는 유료이다.
암호화S3에 저장되는 데이터를 자동으로 암호화할 수 있다.
액세스 제어S3버킷에 대해 권한을 설정할 수 있다.
웹서비스S3버킷을 웹사이트로 사용한다.

Storage Class

S3에서 제공하는 서비스 수준을 의미한다. 모든 스토리지 클래스는 eleven nine 내구성을 보장한다.

  • Standard
    기본 클래스, 이 클래스만 최소 저장 기간에 대한 요금이 없다. 가용성은 99.99% 이다.

  • Intelligent-Tiering
    빈번한 액세스와 간헐적 액세스에 최적화된 두가지 계층에 객체를 저장한다. 어느 쪽에 저장할지는 객체별 모니터링한 결과를 통해 정해지고 이동된다.
    예를 들어 30일간 연속으로 액세스가 없는 객체는 간헐적 액세스 계층으로 이동되고 이 객체에 대한 액세스가 늘어나면 다시 빈번한 액세스 계층으로 이동된다.

  • Infrequent Access
    액세스 빈도가 낮고 용량이 큰 데이터에 적합한 클래스로 Standard 보다 요금이 낮은 대신 액세스 요금이 높다.

    • Standard-infrequent Access
      최소3개 이상의 AZ에 저장된다. 가용성은 99.9% 이다.
    • OneZone-Infrequent Access
      1개의 AZ에만 저장된다. AZ에 문제 발생하면 데이터를 유실할 가능성이 있다.

  • Reduced Redundancy Storage (RRS)
    standard에 비해 이중화 수준을 낮춰 더 낮은 가격으로 제공된다. One AZ이므로 데이터 유실 가능성이 높다.
    스토리지 클래스가 아니라 옵션으로 분류되지만 실질적으로 스토리지 클래스처럼 쓰인다.
    데이터 내구성, 가용성은 99.99%이다.

  • S3 Glacier/ S3 Glacier Deep Archive
    장기간 백업을 고려해 만든 스토리지 클래스이다. 가격이 낮아 대용량 데이터를 낮은 비용으로 보관할 수 있다. 데이터는 Vault 라는 컨테이너에 저장된다. 다시 vault에 저장된 데이터를 읽으려면 S3버킷으로 옮겨야 하는 작업이 필요하다. Glacier Select라는 기능으로 검색도 가능하다. 그러나 vault에 저장된 데이터를 검색하려면 Glacier Select를 사용하게 되므로 요금이 부과된다.

AWS S3 사용 절차

  1. 관리콘솔 -> S3 대시보드 이동
  2. 버킷을 만든다, 버킷명을 지정한다.
  3. 버킷 설정
  4. 파일을 업로드한다.

버킷

  • 버킷은 객체를 담는 컨테이너이고 객체는 파일과 같은 것이다.
  • 버킷은 디렉토리구조가 아니므로 nested 하게 버킷 내부에 버킷을 만들 수 없다.
  • 버킷명은 S3내에서 global unique 한 이름이어야 한다.
  • 버킷은 AWS 계정당 100개만 생성할 수 있다. 버킷1개에 저장할 수 있는 객체 수와 용량은 제한이 없다.

버킷명 명명 규칙

버킷 생성 후 리전과 버킷명을 변경할 수 없기에 신중하게 결정해야 한다.

  • AWS S3의 버킷 명칭은 전 세계적으로 유일해야 한다.
  • 영문 소문자, 숫자, 대시(-), 마침표(.)를 사용할 수 있다.
  • 버킷 이름은 3 ~ 63자 사이여야 하며, 숫자로 시작하거나 마침표(.)나 대시(-)로 끝나면 안 된다.
  • 192.168.5.4 같은 IP 형태면 안된다.
  • s3.amazonaws.com 같은 형태는 사용할 수 없다.
  • DNS label (e.g., "example.com") 같은 형태는 허용되지 않는다.

마침표(.)는 이름에 사용할 수 있지만 SSL 인증과 DNS 관련 문제 발생하는 원인이 될 수 있어 권장되지 않는다.

규칙을 지켜서 명명한 버킷명 예는 아래와 같다.

  • my-awesome-bucket
  • company-name-data
  • project-files-2023
  • app-images-production
  • test-environment-123

로그 외의 액세스 관리 방법

  • 스토리지 클래스 분석
    객체의 액세스 빈도를 분석하는 기능으로 버킷이 어떻게 사용되고 있는지 현황을 확인할 수 있다.
  • 객체 잠금
    객체를 보호하기 위해 잠금으로서 변경을 허용하지 않는다. 객체 삭제, 덮어쓰기, 변조 등을 방지한다. 잠금 기능은 유효기간이 있다. 잠금으로 보호되는 버킷은 Cross-Region Replication2 으로 복사할 수 없다.

교차 리전 복제

Cross-Region Replication은 다른 리전에 버킷에 객체를 비동기적으로 복사하는 것이다. 복사 대상 버킷은 원본 버킷과 소유자가 갖지 않아도 되지만 복제 작업을 수행하기 위해 IAM을 부여해야 한다.

복제에 사용할 두 버킷은 버전관리가 enable 되어 있어야 한다. 같은 리전에 존재하는 버킷은 복제 설정을 할 수 없다.
교차 리전 복제를 사용하면 해외에도 백업이 가능해지므로 한 리전에 disaster가 발생해도 데이터를 유실하지 않고 보존할 수 있다.
해외에 데이터를 옮겨야 한다면 국외 반출이 가능한 데이터인지 주의해야 한다.(회사에서 관리하는 고객 데이터의 경우 더 조심해야 한다.)
