친절한 SQL 튜닝 정리 - 3

친절한 SQL 튜닝 정리 - 3

이 문서는 조시형의 저서 친절한 SQL 튜닝을 개인적으로 참조하기 위해 요약,정리한 것으로 분산된 내용이 많아 책 없이 읽을 수 없는 문서입니다.



3장 인덱스 튜닝

인덱스 ROW ID를 통한 블록 접근은 생각보다 많은 비용이 든다.

인덱스를 스캔하는 이유는 검색조건을 만족하는 소량의 데이터를 인덱스에서 빨리 찾고 거기서 테이블 레코드를 찾아가기 위한 주소값(ROW ID)을 얻으려는 있다.
ROW ID는 데이터파일 번호, 오브젝트 번호, 블록 번호로 구성되어 있지만 포인터같이 물리적으로 바로 접근할 수 있지 않기 때문에 ROW ID는 논리적 주소이다.
INDEX SCAN을 할 때는 리프블록에서 읽은 ROW ID를 분해하고 DBA를 얻는다. TFS 할때는 익스텐트 맵을 통해 읽을 블록들의 DBA 를 얻는다.

블록을 읽을 때 디스크로 접근하기 전에 버퍼캐시부터 찾아보고, 읽고자 하는 DBA를 해시 함수에 입력해 해시 체인을 찾고 버퍼헤더를 찾는다. 버퍼 헤더에는 실제 데이터가 담긴 버퍼 블록의 메모리 위치값을 갖고 있다.
(캐시는 자주사용되는 데이터로 업데이트 되므로, 실제 데이터는 캐시에 매번 다른 위치에 저장된다. )

데이터가 캐시에 있더라도 테이블 레코드를 읽기 위해서는 매번 DBA 해싱과 래치 획득 과정을 반복하고, 서로 다른 프로세스에 의한 동시접근이 발생 할때는 체인 래치와 버퍼 Lock에 대한 경합까지 일어난다. ROW ID를 이용한 테이블 접근은 생각보다 고비용 구조다.

인덱스 클러스터링 팩터

CF 라는 이름으로 불리는 이것은 특정 컬럼을 기준으로 같은 값을 갖는 데이터가 물리적으로 서로 모여 있는 정도를 의미한다.
CF가 높은 컬럼에 생성한 인덱스는 검색 효율이 매우 좋다.

이것은 테이블 액세스량에 비해 블록 I/O가 적게 발생함을 의미하며 이것의 효과는 Buffer Pinning 때문에 생긴다.
버퍼 피닝은 ROW ID로 테이블에 접근할때 래치 획득과 체인 스캔 과정을 거쳐 찾아간 블록에 대한 메모리 주소를 바로 해제하지 않고 유지하는 것이다.
이렇게 메모리 주소를 바로 해제하지 않은 상태로 다음 인덱스 레코드를 읽었을 때 이전에 읽은 블록을 가리킨다면 래치 획득과 체인 스캔을 생략하고 이전의 해제 되지 않은 메모리 주소를 통해 테이블 블록을 읽을 수 있다.

인덱스 손익분기점

테이블의 일정한 크기를 넘어서면 인덱스 ROWID를 이용한 테이블 접근은 테이블 전체를 스캔하는 것보다 느려진다.

Index Range Scan과 TFS를 수행한 것을 X축을 추출건수로, Y축을 소요시간으로 했을 때의 2차원 라인그래프로 표현하면 TFS는 추출 건수가 증가함과 상관없이 소요시간은 변하지 않으므로 x = y 이며 IRS는 추출 건수가 증가함에 따라 소요시간이 점차 증가하여 y = nx 형태의 선이 나타나게 된다.

이때 두 선이 교차하는 지점을 손익분기점이라고 칭한다. 즉, IRS가 일정한 량의 테이블 액세스량을 넘으면 TFS가 더 빠르다는 것이다.

그 차이는 IRS는 인덱스 ROW ID를 사용한 랜덤 액세스 방식을 사용하고, 또한 SingleBlock I/O 를 사용하기 때문에 처리율(Throughput) 측면에서는 Sequantial Access 와 MultiBlock I/O 를 사용하는 TFS보다 낮을 수 밖에 없다.

인덱스 손익분기점은 일반적으로 5~20% 낮은 수준(10만 ~ 100만 ROW 규모 테이블) 에서 결정되며 CF에 따라 크게 변동한다. (5% 미만 ~ 90%)

143p 온라인 vs 배치

온라인에서는 소량 데이터를 읽고 갱신하므로 인덱스를 효과적으로 활용하는 것이 중요하고 조인방식도 이때는 index를 사용하는 Nested Loop 조인을 대부분 사용한다.

배치 프로그램은 전체범위 처리 기준으로 튜닝해야 하므로 TFS와 Hash Join을 사용한다.

예제로서 온라인을 보여줄 때는 아무런 힌트도 없었지만 랜덤 액세스가 많이 나오는 쿼리에서는 TFS와 일부 IFFS, 해시 조인을 사용하는 힌트를 부여하여 튜닝을 하였고 이 튜닝된 쿼리에서 동일한 테이블을 두번 읽는 비효율이 발생하여 추가적으로 window 함수를 사용하였다.

145p 테이블 액세스 최소화를 위해 가장 일반적인 튜닝 기법은 기존 인덱스에 컬럼을 추가하는 것이다.

select /*+ index(emp emp_x01) */
FROM EMP
WHERE dept_no = 30
and sal >= 2000

emp_x01은 (dept_no, job) 으로 구성되어 있어서 sal을 인덱스에서 검색할 수 없기에 dept_no 로 인덱스로 찾은 뒤 나온 블록을 모두 비교한다.

이때 (dept_no, sal) 로 변경하면 좋겠지만 실제 DB 운영환경에서 인덱스 구성은 변경하기가 쉽지 않다.
기존 인덱스를 구성한 이유가 있기 때문이며, 또한 인덱스를 새로 추가할 수도 있지만 많은 부분이 고려되지 않은 인덱스 추가는
인덱스 관리비용을 증가시킨다(DML 부하에 따른 트랜잭션 성능 저하)가 발생할 수 있다.

이때 (dept_no, job, sal) 로 추가하게 되면 기존 쿼리에 영향을 미치지 않고, 성능을 개선할 수 있다.
세번째 인덱스이므로 전체 인덱스 스캔량은 추가하기 전과 동일하겠지만 인덱스에 sal이 추가되어 값을 비교할 수 있기 때문에
테이블 랜덤 액세스를 줄일 수 있는 것이다. 약간의 인덱스 관리 비용이 증가하겠지만 새 인덱스를 추가하는 것보다는 나은 방법이다.

include 인덱스 MSSQL Server 2005에 추가된 기능으로 인덱스 키 외에 컬럼을 리프 블록에 저장하는 기능이다. 아래와 같이 사용한다.

create index emp_x01 on emp (dept_no) include (sal)

일반 인덱스는 sal 컬럼을 루트와 브랜치 블록 모두에 저장하여 수직적 탐색에 사용할 수 있고
include index는 리프블록에만 사용한다. 일반 인덱스가 성능 측면에서는 더 좋지만
include 인덱스는 더 적은 비용으로 리프 블록에서 sal을 비교하여 랜덤 액세스 횟수를 줄이는 용도로 사용되는 것이다.

create index emp_x02 on emp (dept_no, sal)

Index Organized Table

일반적으로 테이블을 하나 만들고, 인덱스를 추가하는 방법과 다르게 IOT는 랜덤 액세스가 아예 발생하지 않도록 테이블을 인덱스 구조로 생성하는 방법이다. MS-SQL 에서는 Clustered 인덱스라고 부른다.

이 테이블은 일반 인덱스 리프 블록에 ROW ID가 아니라 실제 데이터가 연결된 것과 같은 구조를 가지게 되어 리프 블록이 곧 데이터 블록이 되는 것이다.

IOT에 빗대어 일반 테이블은 heap table 이라고 부른다.
테이블 생성 시 IOT는 create table ... organization index 라고 사용하고
일반 테이블은 기본값이므로 보통 생략하게 되지만 create table ... organization heap 이라고 명시하기 때문이다.

일반 heap 구조 테이블은 데이터를 입력할 때 랜덤 방식을 사용하게 된다.
Freelist로부터 할당받은 블록에 정해진 순서 없이 데이터를 입력하는데 IOT는 정렬된 상태를 유지하며 데이터를 입력하게 된다.
IOT는 정렬되어 동작하므로 CF가 매우 높게 나타난다.

클러스터 테이블

클러스터 테이블은 인덱스 클러스터와 해시 클러스터 두 가지가 있다.

클러스터 테이블은 테이블에 속한 하나의 키를 기준으로 값이 같은 레코드를 하나의 블록에 모아서 저장하는 구조다.
기준이 되는 키를 클러스터 키라고 부르며 한 블록 크기를 넘어가는 레코드는 새로운 블록을 할당한 후 클러스터 체인으로 연결한다.

클러스터 테이블은 여러가지 테이블 레코드를 같은 블록에 저장할 수도 있으며 이것을 다중 테이블 클러스터 라고 부른다.
일반 테이블을 사용할때는 하나의 블록을 여러가지 테이블이 공유할 수 없다.

인덱스 클러스터 테이블은 아래의 절차를 통해 생성한다.

  1. 클러스터 객체 생성
CREATE CLUSTER C_DEPT# ( DETPNO NUMBER(2) ) INDEX;
  1. 클러스터 인덱스 생성
CREATE INDEX C_DEPT#_IDX ON CLUSTER C_DEPT#;
  1. 클러스터 인덱스를 사용한 테이블 생성
CREATE TABLE DEPT (
    DEPTNO NUMBER(2) NOT NULL,
    DNAME VARCHAR2(14) NOT NULL,
    LOC VARCHAR2(13)
) CLUSTER C_DEPT#( DEPTNO );

클러스터 인덱스는 일반 B*Tree 처럼 인덱스 구조이며 일반 테이블의 인덱스 레코드는 테이블 레코드와 1:1 대응관계를 갖지만
클러스터 인덱스는 테이블 레코드와 1:M 관계를 가진다. (하나의 클러스터 인덱스가 여러개의 블록을 가리킨다) 그래서 클러스터 인덱스의 키값은 항상 Unique 하다.

이런 구조는 클러스터 인덱스를 스캔할 때 랜덤 액세스를 값 하나당 한번씩만 발생하고
이후로 클러스터( 같은 클러스터 키를 가진 블록들) 에 도착했을 때는 시퀀셜 액세스를 하기에 넓은 범위를 읽어도 비효율이 없어지게 된다.

해시 클러스터는 클러스터 인덱스 대신 해시 함수로 클러스터를 구분한다는 것만 다르다.

해시 클러스터 테이블은 아래의 절차를 통해 생성한다.

  1. 클러스터 객체 생성
CREATE CLUSTER C_DEPT# ( DETPNO NUMBER(2) ) HASHKEYS 4;
  1. 클러스터를 사용한 테이블 생성
CREATE TABLE DEPT (
    DEPTNO NUMBER(2) NOT NULL,
    DNAME VARCHAR2(14) NOT NULL,
    LOC VARCHAR2(13)
) CLUSTER C_DEPT#( DEPTNO );

부분범위 처리

DB는 1억개의 컬럼을 모두 가져온다고 해도 데이터를 모두 읽어 클라이언트로 전송하지 않고 일정량(Array Size)만 전송하고 멈추기 때문이다.
다음 데이터는 클라이언트에서 Fetch Call을 할때 전송하게 된다. 이렇게 처리하는 것을 부분범위 처리라고 부른다.

Fetch Call 마다 얼마나 가져올 것인지는 클라이언트에서 결정한다. Array Size 는 레코드의 갯수를 말하는 것이다.
JDBC에서는 기본값이 10이다. 가져오려는 전체 데이터가 100개이면 10번의 Fetch Call이 필요하다.

정렬 조건이 있을 때는 전체 범위 처리가 되지만 정렬에 사용된 컬럼이 인덱스 선두 컬럼이면 부분 범위 처리가 가능하다.

Array Size 는 작으면 반응속도가 빠르지만 처리율을 기준으로 했을 때는 값을 크게 설정해야 한다.

OLTP 환경에서 부분범위 처리에 의한 성능개선 원리

SELECT *
FROM 게시판
WHERE 게시판구분코드 = 'A'
ORDER BY 등록일시 DESC

이 쿼리에서 조건절을 인덱스가 바로 검색할 수 있는 형태(게시판구분코드 단일 조건 또는 (게시판구분코드, 등록일시) )
로 사용하지 않으면 소트 연산을 생략할 수없어서 많은 량의 테이블 랜덤 액세스가 발생한다.

배치 I/O

일반적으로 쿼리는 인덱스를 이용해 테이블을 접근하다가 버퍼 캐시에서 찾아본 후 디스크에서 블록을 읽는다.
배치 I/O는 이때 테이블 블록에 대한 디스크 I/O call을 미뤘다가 읽을 블록이 일정량 쌓이면 한꺼번에 처리하는 방법이다.

oracle 11g 에서 NL 조인을 INNER 테이블 액세스에만 작동하며 12c에서는 ROW ID로 테이블을 접근하는 모든 부분에서 작동한다.
배치 I/O는 정렬이 보장되지 않기 때문에 소트연산이 필요없는 경우에만 사용해야 한다.

ORDER BY 를 사용할 경우에 배치 I/O는 옵티마이저에 의해 선택되는 경우가 많은데, 이런 쿼리를 관리할 수 없을 경우 시스템 차원에서 비활성화하는 경우가 있다.

인덱스 스캔 효율화

173 ~ 184 페이지까지 여러가지 예제를 보여주며 스캔의 효율성을 비교하고 있다.
특히 조건절에서 인덱스 구성에 따라서, 또한 연산자를 변경해가면서 예제를 설명해주고 있다.
184절에는 간단한 결론을 도출한다.

조건절에서 인덱스 선행 컬럼이 조건절이 없거나 등치 연산자가 아니면 스캔 과정에서 비효율이 발생한다.

인덱스가 (A,B,C,D) 일때 순서대로 조건절에 A = :a AND B = :b AND C = :c AND D = :d 라고 사용할 때
B를 빠뜨리고 A = :a AND C = :c AND D = :d 이렇게 사용다던지 하는 것,
즉 인덱스 컬럼 중 어떤 컬럼이던지 앞에 있는 컬럼이 없이 나머지 인덱스 컬럼들이 조건절에 선언된다면 비효율이 발생한다는 뜻이다.

인덱스 스캔 효율성 측정

SQL 트레이스에서 INDEX SCAN 되는 행을 관찰하면 ROW 컬럼에는 표시되는 숫자는 인덱스를 스캔하고 가져온 레코드 갯수(irs_record_count)가 표시된다.

RANGE 스캔의 cr은 읽은 블록의 갯수다.
블록과 레코드는 일대다 관계이므로 cr에 블록당 500개의 레코드가 있다고 가정하고 cr * 500을 해서 나온 숫자가 rows 에 표시된 레코드 갯수와 비교하면 된다.

irs_record_count / cr * 500

이 값이 1에 가까울수록 SQL 스캔효율이 높다고 할 수 있다.

185p( 필수개념이므로 확실하게 이해할 것)

액세스 조건, 필터조건

먼저 책에서 보여준 그림을 떠올려보자

삼각형으로 된 인덱스 그림이 있고 삼각형 인덱스에는 화살표가 수직적 탐색과 수평적 탐색으로 표현되는 화살표가 표시되어 있다.
이것을 액세스 조건이라고 부르고 있다. 우리가 쿼리를 한다면 그 쿼리의 조건절에 의해 어떤 범위에서 인덱스가 스캔할 것인지가 결정될것이다.

설명1
액세스 조건은 인덱스 스캔 범위를 결정하는 조건절을 말하는 것이다.

설명2
액세스 조건은 수직적 탐색을 통해 스캔 시작점을 결정하는 데 영향을 미치고, 수평적 탐색을 통해 리프 블록을 스캔하다가 어디서 멈출지를 결정하는데 영향을 미치는 조건절이다.

그리고 삼각형 인덱스 그림 밑에는 실제 인덱스 리프블록이 갖고 있는 테이블이 일부가 블록으로 표시되어 있으며 수평적 탐색에서 가져올 수 있는 ROW ID와 연결되는 화살표가 표시된다.

이것이 인덱스 필터 조건이라고 부르고 있다. 인덱스 필터 조건은 리프 블록을 스캔후 가져온 ROW ID를 갖고 테이블로 액세스 할 것인지를 결정하는 조건절이다.

테이블 필터 조건은 테이블에 접근 후 가져온 데이터를 쿼리 수행 다음 단계로 전달하거나 최종 결과집합에 포함할지를 결정한다.

186p
옵티마이저의 비용 계산 원리
이 공식은 인덱스에서 테이블 액세스까지의 절차를 잘 떠올리게 해준다.

인덱스 수직적 탐색 비용 + 인덱스 수평적 탐색 비용 + 테이블 랜덤 액세스 비용
=
인덱스 루트와 브랜치 레벨에서 읽는 블록 수 +
인덱스 리프 블록을 스캔하는 과정에서 읽는 블록 수 +
테이블 액세스 과정에 읽는 블록 수

188p

(184p에서 내린 결론에서 추가적으로 생각하자)

조건절에 선언된 선행 컬럼이 모두 '=' 인 상태에서 첫번째 나타나는 범위검색 조건까지만 만족하는 레코드는 모두 연속해서 모여 있지만
그 이하 조건까지 만족하는 레코드는 비교 연산자 종류에 상관없이 흩어진다.
(흩어져 있다는 것은 수평적 탐색시 필요하지 않은 조건까지 모두 스캔해야 한다는 것이다.)

이 페이지에서는 또 액세스 조건과 필터 조건을 결정하는 공식같은 것을 말하고 있다.

'선행컬럼이 모두 '=' 인 상태에서 첫번째 나타나는 범위검색 조건이 인덱스 스캔 범위를 결정한다'

이 공식은 실제 옵티마이저가 결정하는 액세스 조건과 필터 조건과는 다르다.
그래서 189p에서는 실제 옵티마이저 보여주는 액세스, 필터 조건들과 왜 다른지를 설명한다.

오토 트레이스의 Predicate Information 에서는 액세스, 필터 조건들을 보여준다.

책에서는 Predicate Information 에 나타나는 조건들은 조금이라도 스캔량을 줄이는 데 도움이 된다면 액세스 조건이라고 판단하며,
그렇지 않다면 필터조건이라고 나오게 된다는 것이다.

Predicate Information 필터 조건으로 나타나는 경우는 다음과 같다.

  1. 조건절에서 좌변 컬럼을 가공했을 경우
  2. like 사용시 왼쪽에 %를 포함했을 경우
  3. 같은 컬럼 조건절이 2개 이상일 때 인덱스 액세스 조건으로 선택되지 못한 조건절
  4. OR Expansion 또는 INLIST Iterator로 선택되지 못한 OR 또는 IN 조건절

결론:
알고리즘에서 액세스 조건과 필터 조건을 구분하는 기준이 있는데도 불구하고 새로 정의한 이유는 스캔량을 미미하게 줄이는 것이 액세스 조건으로 판단하는 것은 오히려 전체적인 DB 튜닝 업무에서는 필터 조건으로 구분되는 것이 더 낫다는 판단을 한 것으로 보인다.
