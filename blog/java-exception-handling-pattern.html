java exception handling pattern

AntiPattern 1: 모든 Exception catch 하기

catch block 에서 예외를 가장 상위 클래스인 Exception으로 catch하면 모든 예외를 catch 하게 된다. 이것은 중요한 오류가 숨기게 되는 것으로 문제를 디버깅하기가 더 어려워지게 만든다.

try {
// 예외를 발생시킬 수 있는 코드
} catch (Exception e) {
// 모든 익셉션을 캐치하는 코드는 bad practice 이다.
}

best practice

모든 exception을 catch하는 대신 발생할 것으로 예상되는 특정한 exceptions만 catch 해야 한다.

try {
// 예외를 발생시킬 수 있는 코드
} catch (FileNotFoundException e) {
// FileNotFoundException 처리 코드
} catch (IOException e) {
// IOException 처리 코드
}


AntiPattern 2: exception 무시하기

exception을 catch 하고 아무것도 하지 않는 것은 발생한 문제를 숨기는 것과 같다.

try {
    // exception을 throw할 수 있는 코드
} catch (IOException e) {
    // catch 후 아무것도 하지 않기
}

best practice

세부적인 exception 타입을 반환하여 어떤 에러가 발생했는지 전달해야 한다.

try {
    // exception을 throw할 수 있는 코드
} catch (IOException e) {
    // exception을 로깅하기
    logger.error("Error occurred while reading file", e);
}


AntiPattern 3: 흐름제어를 위해 try catch를 사용하기

try-catch 문법은 일반적인 흐름제어를 위해 사용되어서는 안된다.

public class Account {
    private double balance;

    public void withdraw(double amount) throws InsufficientFundsException {
        if (balance < amount) {
            throw new InsufficientFundsException("Insufficient funds");
        }
        balance -= amount;
    }
}

public void someMethod() {
    Account account = new Account();

    try {
        account.withdraw(100);
    } catch (InsufficientFundsException e) {
        // deposit the account and try again
        account.deposit(100);
        account.withdraw(100);
    }
}

best practice

위 코드 같은 경우에는 try-catch 대신 조건문을 사용하여 처리해야 한다.

public class Account {
    public enum WithdrawalStatus {
        SUCCESS,
        INSUFFICIENT_FUNDS
    }

    private double balance; //잔액

    public WithdrawalStatus withdraw(double amount) {
        if (balance < amount) {
            return WithdrawalStatus.INSUFFICIENT_FUNDS;
        }
        balance -= amount;
        return WithdrawalStatus.SUCCESS;
    }

}

public void someMethod() {
    Account account = new Account();

    Account.WithdrawalStatus status = account.withdraw(100);
    if (status == Account.WithdrawalStatus.INSUFFICIENT_FUNDS) {
        // 금액을 입금하고 다시 인출
        account.deposite(100);
        account.withdraw(100);
    }
}


AntiPattern 4: 과도한 checked exception 사용

checked exception을 사용하는 것은 유용할 수 있지만 모든 경우에 사용하는 것은 코드를 복잡하고 유지보수하기 어렵게 만들 수 있다. 아래의 경우에서는 또한 예외 처리시 의미있는 형태로 다루고 있지도 않다.

public class CustomCheckedException extends Exception {
    public CustomCheckedException(String message) {
        super(message);
    }
}

public void processFile(String filename) throws CustomCheckedException {
    if (filename == null) {
        throw new CustomCheckedException("Filename cannot be null");
    }
    // CustomCheckedException을 발생시킬 수 있는 다른 코드
}

public void someMethod() {
    try {
        processFile(null);
    } catch (CustomCheckedException e) {
        System.out.println("Error: " + e.getMessage());
        // 오류를 복구하는 의미있는 형태가 아니다
    }
}

best practice

recover가 불가능한 상황이나 프로그래밍 에러는 unchecked exception 을 사용하는 것이 더 좋은 방법이다.
unchecked exception 를 throw하는 메서드의 사용자는 명시적으로 handle 하지 않아도 되므로 덜 복잡해지고 이것은 메서드의 사용자가 의미 있는 형태로 복구할 수 없거나 수정되어야 하는 프로그래밍 오류로 인해 발생하는 오류에서 특히 유용하다.

public class CustomUncheckedException extends RuntimeException {
    public CustomUncheckedException(String message) {
        super(message);
    }
}

public void processFile(String filename) {
    if (filename == null) {
        throw new CustomUncheckedException("Filename cannot be null");
    }
    // CustomUncheckedException 발생시킬 수 있는 다른 코드
}

public void someMethod() {
    String filename = null;
    try {
        processFile(filename);
    } catch (CustomUncheckedException e) {
        System.out.println("Error: " + e.getMessage());
        // 의미있는 형태: 기본값으로 이름을 set하여 오류에서 recover하는 코드
        String defaultFilename = "default.txt";
        System.out.println("Using default filename: " + defaultFilename);
        processFile(defaultFilename);
    }
}


AntiPattern5: exception catch 후 컨텍스트 추가 없이 다시 throw 하기

exception을 catch 하고 나서 해당 exception에 컨텍스트(맥락) 정보 추가 없이 다시 throw하는 것은 디버깅 정보의 손실이다.

try {
    // exception을 throw할 수 있는 코드
} catch (IOException e) {
    throw e; // 컨텍스트 추가 없이 다시 throw 하기
}

best practice

exception을 상위로 전달하기 위해 다시 throw 할 때 디버깅에 도움이 될 수 있는 context를 추가한 exception을 throw 해야 한다.

try {
    // exception을 throw할 수 있는 코드
} catch (IOException e) {
    throw new CustomException("Failed to process file", e);
}


AntiPattern6: 일반 exception 반환하기

일반 exception throw 하는 것(예> throw new Exception()) 은 오류에 대한 충분한 정보를 제공하지 못한다.

public void writeFile(String content) throws Exception {
    if (content == null) {
        throw new Exception();
    }
    // 파일 쓰기 코드
}

best practice

특정 exception 타입을 사용하여 오류의 특성을 전달해야 한다.

public void writeFile(String content) throws FileNotFoundException {
    if (content == null) {
        throw new FileNotFoundException("Content not found");
    }
    // 파일 쓰기 코드
}


AntiPattern7: exception 대신 에러 코드를 사용하는 것

오류를 다루기 위해 에러코드를 사용하는 것은 코드를 이해하거나 유지보수하기 어렵게 만든다. 대신 예외를 사용하여 오류를 처리하는 명확하고 일관된 방법을 제공해야 한다.

아래 example 에서 processFile 메서드는 에러코드를 사용하여 다양한 오류 조건을 나타낸다. 호출하는 메서드(someMethod)는 이러한 에러코드를 확인하고 처리해야 하므로 오류 코드가 더 추가되면 번거롭고 오류가 발생하기 쉽다.

public int processFile(String filename) {
    if (filename == null) {
        return -1; // 파일이름이 없는 경우 에러 코드 반환
    }

    if (!fileExists(filename)) {
        return -2; // 존재하지 않는 파일에 대한 에러 코드 반환
    }

    return result;
}

public void someMethod() {
    int result = processFile("example.txt");
    if (result == -1) {
        System.out.println("Error: Filename is null");
    } else if (result == -2) {
        System.out.println("Error: File does not exist");
    } else {
        System.out.println("File processed successfully");
    }
}

best practice

public void processFile(String filename) throws FileNotFoundException, IllegalArgumentException {
    if (filename == null) {
        throw new IllegalArgumentException("Filename cannot be null");
    }

    if (!fileExists(filename)) {
        throw new FileNotFoundException("File not found");
    }

    // 파일 처리 코드
}

public void someMethod() {
    try {
        processFile("example.txt");
        System.out.println("File processed successfully");
    } catch (IllegalArgumentException e) {
        System.out.println("Error: " + e.getMessage());
    } catch (FileNotFoundException e) {
        System.out.println("Error: " + e.getMessage());
    }
}


AntiPattern8: 로깅과 exception throw 같이 하기

exception을 로깅한 다음 throw 하면 호출 메서드(calling method)가 exception도 로깅할 수 있으므로 로그 항목이 중복될 수도 있다.

try {
    // exception을 throw할 수 있는 코드
} catch (IOException e) {
    // 로깅과 throw를 같이 하는 코드
    logger.error("Error occurred", e);
    throw e;
}

best practice

로깅 및 exception return 중 해야한다면 둘 중 하나만 해야 한다.

try {
    // exception을 throw할 수 있는 코드
} catch (IOException e) {
    // exception을 로깅하지만 반환하지는 않는 코드
    logger.error("Error occurred while reading file", e);
    handleFileError();
}
