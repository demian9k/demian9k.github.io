도메인 주도 설계로 시작하는 마이크로 서비스 개발 - 4, 5장

이 문서는 한정헌,유해식,최은정,이주영의 저서 도메인 주도 설계로 시작하는 마이크로 서비스 개발을 정리한 문서입니다.



4장 마이크로서비스와 애자일 프로세스

이 장은 이 책의 목적인 DDD를 사용한 마이크로서비스 구현을 위해 agile 프로세스가 적합하다고 말하면서 간략하게 소개한다.

이미지 출처: pm-partners

4.2.1 점진/반복적인 스크럼 생명주기

스크럼
애자일 개발방법론 중 하나

스프린트
팀이 일정량의 작업을 완료하는 시간이 정해진 짧은 기간(1~4주)

스크럼 팀
하나의 서비스를 만들기 위한 architect, domain expert, software, QA, design 등 구성원 모임 스크럼 마스터가 팀의 리더로 협업을 촉진하고 팀의 장애물을 제거한다.

스크럼 미팅 Daily Scrum
매일 아침 스탠드업 미팅을 통해 현재 하고 있는 일을 스크럼 팀과 공유

스프린트 계획 Sprint Planning Meeting
요구사항을 backlog(일감 목록)로 남긴다. 스프린트 횟수를 결정하고 backlog를 각 스프린트에 배분한다.
스프린트가 끝나면 백로그 완료 일감을 기준으로 팀을 생산성을 측정한다. 이것을 속도라고 한다.
이 속도를 기준으로 팀의 스프린트 계획이 세워진다. continuous planning 이라고 부르는 절차로 전통적인 폭포수 모델에 비해 상황에 따라 적응적인 계획이다.
스크럼에서는 프로젝트의 일정, 비용, 품질이 고정시키고 각 진행상황에 따라 일의 범위가 조정된다.
반면 기존의 전통적인 개발프로세스는 범위는 고정되어 있고, 일정, 비용, 품질이 가변적이다.

평가 Sprint Review
시연 및 검토하여 백로그들이 모두 구현되고 요건을 만족하는지 확인한다. 피드백을 받아 다음 스프린트에 반영한다.

회고 Sprint Retrospectives
스크럼 팀이 프로젝트를 돌아보는 과정이다. 좋았던 점과 안 좋았던 점을 논의하고 개선점을 찾아 다음 스프린트에 적용할 수 있다.

4.2.2 아키텍처 정의와 마이크로서비스 도출

아키텍처 정의
내부 아키텍처, 외부 아키텍처를 정의하는 절차 외부 아키텍처는 언제나 바뀔 수 있는 것으로 간주한다. (Clean architecture, 외부 영역은 언제나 세부사항이다.)

마이크로서비스 도출
스크럼 팀이 DDD의 전략적 설계 기법을 사용해 마이크로서비스를 도출한다.

하나의 도메인을 하위 도메인 요소들로 분해하여 어느 하위 도메인이 마이크로서비스가 될 것인지를 결정한다. 설계가 끝나면 스크럼 팀에 분배하여 스크럼을 진행한다.

4.2.3 스프린트 내 개발 공정

백엔드 설계 및 개발

  1. API 설계를 먼저 진행하여 프론트엔드에 제공한다.
  2. 도메인 모델과 데이터 모델을 설계한다.
    DDD는 OOAD 에서의 UML같이 정형화된 모델 없이 화이트보드나 포스트잇등의 간단한 도구로 작성하며 설계한 뒤 코드로 구현한다.

프론트엔드 개발 및 CI/CD 절차는 통상적인 절차와 같으므로 생략한다.




5장 마이크로서비스 설계

5.1 마이크로서비스를 도출하는 방법

5.1.1 비즈니스 능력에 근거한 도출

비즈니스 능력(capability)에 따라 서비스로 식별하는 방법
각 도메인에서 이미 정해진 조직과 부서에 따라 단순하게 도출하는 방법이다.

이미지 출처: SK C&C TECH BLOG

쉽고 직관적이지만 이후 다룰 DDD 기반 방법에 비교했을 때 단점이 있다.

  • 서비스간 관계 파악 어려움
  • 서비스 구체기능과 연관된 서비스가 관리할 독립적 데이터 식별 어려움

5.1.2 DDD의 Bounded Context 를 사용한 도출

MSA에서 database per service 에 따라 각 데이터는 직접 접근하지 않는다. 그래서 전체 데이터 모델 중에 어떤 데이터를 각 MSVC 가 소유해야 하는지를 식별하는 것이 중요하다.

DDD의 전략적 설계에서 bounded context는 비즈니스 응집성으로 구분한 개념으로 단일 마이크로서비스로 식별하는 단위가 된다.

Bounded Context를 사용해서 설계하면, 데이터를 기능과 분리하지 않고 sub domain에 속한 도메인 모델로 정의한다.

아래 그림엔 서브 도메인은 보이지만 그에 속한 도메인 모델은 보이지 않는다. 하지만 서브도메인과 bounded context의 관계, 그리고 어떻게 마이크로서비스로 도출되는지 이해하는 데 도움이 된다.

이미지 출처: SK C&C TECH BLOG

마이크로서비스의 적정 크기
논란이 많은 주제이지만 일반적으로 코드의 라인 수 같은 양으로 판단하지 않고 업무 맥락에서 정성적으로 판단한다. 이 판단 기준은 자율적으로 수행 가능한 업무의 단위, 개념의 일관성, 기능의 응집성 등을 말하는 것이다. 이것은 비즈니스 도메인, 조직의 성숙도에 따라 상대적으로 달라진다.

5.2 DDD로 설계하기

클라우드에서의 MSA 적용을 통해 얻을 수 있는 독립적 개선과 배포, 장애 격리 및 장애 발생시 빠른 대응을 위해 마이크로서비스를 응집성 있게 식별하는 것이 중요하다.

전략적 설계는 마이크로서비스의 내부 구조를 설계하고, 상세한 내용은 전술적 설계법을 사용한다. 이 방법들은 빠른 기술변화에도 유연한 애플리케이션을 설계할 수 있도록 돕는다.

5.3 DDD의 전략적 설계

DDD는 하나의 큰 도메인(전체 애플리케이션) 을 전략적으로 중요한 것들을 찾아 도메인을 여러개로 나누고 각 도메인별로 개별 기능을 해결하는 방법을 기본으로 삼는다.

각 나눠진 도메인들은 논리적으로 구분되는 여러개의 하위 영역으로 서브도메인이라고 부른다.

서브 도메인로의 분할은 소프트웨어에서 어떤 문제를 해결하기 위한 영역으로 하나의 큰 문제를 여러개의 작은 문제로 나눠서 해결하는 접근법으로 Divide and Conquer approach 와 그 맥락을 같이 한다.

서브 도메인은 3가지 유형으로 나뉜다.

핵심 서브도메인(Core Domain)

다른 경쟁기업과 차별화를 만들 비즈니스 영역으로 기업의 프로젝트에서 여러 관심사 중 높은 우선순위를 갖고, 소프트웨어 개발에서도 전략적으로 가장 큰 투자가 필요한 영역이다.

ex) 아마존의 추천시스템, 쿠팡의 로켓배송 등

지원 서브도메인(Supporting Subdomain)

비즈니스에 필수적이지만 핵심은 아닌 부분으로 볼 수 있다. 그러나 핵심 도메인을 성공시키기 위해서 반드시 필요한 영역으로 핵심 도메인에 이어 2번째로 중요하다.

일반 서브도메인(Generic Subdomain)

비즈니스적으로 특별하진 않지만 전체 비즈니스 솔루션에 필요한 부분으로 기존 제품으로 구매해서 대체할 수 있는 영역이다.

Ubiquitous Language, bounded context

고객이나 현업 담당자가 사용하는 비즈니스 언어와 소프트웨어 개발 과정에서 사용하는 언어가 같은 의미에도 다르게 사용되는 경우가 많다. 그리고 같은 발음의 단어라도 특정한 맥락에 따라 의미가 다르게 해석될 수 있는 경우도 많다.

DDD 에서는 언어의 이러한 사용으로 인해 발생하는 문제를 방지하기 위해, 특정 도메인에서 의도를 명확히 반영하고 핵심 개념을 잘 전달 할 수 있는 언어를 유비쿼터스 언어로 정의한다.

유비쿼터스 언어는 일반적으로 데이터모델링에서 사용하는 표준 단어/용어 사전과 다른 개념으로 세부적인 도메인에서 특정 업무와 관련된 사람들 사이에서 자율적으로 정의되고 통용되는 언어이다.

예를 들어 결제 도메인에서의 고객, 배송 도메인에서의 고객은 의미가 다를 수 있다. 결제 도메인에서 고객은 금전을 지급하고 구매한 결제자(payer)이고 배송 도메인에서는 물건을 받을 수취인(recipient) 으로 각각 다르게 정의할 수 있다.

도메인 모델은 특정 비즈니스 맥락에서 통용되는 개념들의 관계를 정의한 모형이다. 그래서 잘 설계된 도메인 모델을 보면 비즈니스를 이해할 수 있다.

여러가지 도메인 모델을 설계하다 보면 경계가 생긴다. 유비쿼터스 언어들의 개념이 상이한 경계가 도메인의 경계인 Bounded Context이다.

도메인 모델은 도메인과 관련된 업무를 수행하는 제품 책임자, 도메인 전문가, 개발자를 비롯한 구성원들이 업무를 이해하는 기본 모형이 된다. 도메인 모델에 정의된 언어로 업무 협의를 진행하고, 개념을 발전시켜 나간다. 또한 이 모델의 언어를 그대로 사용해 설계 산출물을 표현하고 소스코드에서도 그대로 사용한다.

새로운 구성원이 팀에 합류하더라도 소스코드의 도메인 모델로 도메인 전문가와 무리없이 소통할 수 있다면 도메인과 서비스에 대한 비전을 공유할 수 있고 서비스는 지속적으로 민첩하게 개발, 유지될 수 있다.

유비쿼터스 언어를 바운디드 컨텍스트, 도메인 모델의 관계를 고려하여 다음과 같이 정의할 수 있다.

바운디드 컨텍스트 내의 도메인 주요 개념을 표현하기 위해 도메인 내에서 공통으로 사용하는 언어

Context Mapping

Bounded Context 를 식별할 때 컨텍스트는 내부적으로 응집성이 높고 다른 컨텍스트와 의존관계가 낮아야 한다는 원칙 아래에서 설계한다.

그것은 컨텍스트 사이에 아무런 관계가 없다는 의미는 아니다. 여러개의 바운디드 컨텍스트로 식별하면 비즈니스 로직 연결을 위해 컨텍스트간에 연결되어야 하는 경우가 발생한다.

이러한 연결하는 것을 DDD의 Context Mapping 이라고 부르며 두 컨텍스트 사이의 선을 그려서 표시한다. 컨텍스트 사이 매핑 관계를 표시한 다이어그램을 컨텍스트 맵이라고 한다. 컨텍스트 맵은 다수의 매핑으로 구성되며 여러가지 매핑 관계가 있다.

주요 컨텍스트 매핑 관계

공유 커널(Shared Kernel)

공유커널은 컨텍스트 경계 사이에 공통적인 모델을 공유하는 관계이다. 공유커널은 두개 이상의 팀에서 작지만 공통의 모델을 공유하는 관계를 나타낸다. 각 팀은 공유하는 모델에 서로 합의해야 한다.

보통 공통 라이브러리등이 될 수 있는데 이 부분이 변경되며 여러 관련 컨텍스트에 영향을 미치므로 공유하는 모델의 코드 빌드를 관리하고 테스트하는 것은 한 팀에 맡아 수행하도록 해야 한다.

이미지 출처: SK C&C TECH BLOG

소비자와 공급자(Customer-Supplier)

공급하는 컨텍스트는 upstream(상류) 로 소비하는 컨텍스트는 downstream(하류) 로 표시한다. 데이터의 흐름은 상류에서 하류로 흐른다. 반대는 가능하지 않으며, 상류의 변화가 생기면 하류에서 변화를 따른다. 공급자는 소비자가 원하는 기능을 제공해야 한다.

이미지 출처: SK C&C TECH BLOG

준수자(Conformist)

준수자는 소비자와 공급자 매핑에서 상류 팀이 하류팀의 요구를 지원하지 않거나 못하는 경우에 이 매핑을 사용한다. 하류팀은 상류 팀이 제공하는 모델을 그대로 사용하는 것이다.

충돌 방지 계층(Anti-Corruption Layer)

하류 팀이 상류팀의 모델에 영향을 받을 때 하류 팀의 고유 모델을 지키기 위한 번역 계층을 만드는 것이다. 번역 계층은 상류 모델의 변경 없이 하위 모델과 통합하기 위해 데이터를 변환하는 것은 구현한 것이다.

레거시 시스템을 변경 없이 MSVC 아키텍처로 통합할 때 유용하게 사용할 수 있다.

이미지 출처: SK C&C TECH BLOG

공개 호스트 서비스(Open Host Service)

바운디드 컨텍스트에 대한 접근을 제공하는 프로토콜이나 인터페이스를 정의한다. 이 프로토콜은 하류의 컨텍스트가 상위 컨텍스트에서 제공하는 기능을 용이하게 사용할 수 있도록 공개돼 있다. 보통 다른 컨텍스트에서 사용할 수 잇는 공유된 API를 제공하기 위해 사용한다.

이미지 출처: SK C&C TECH BLOG

발행된 언어(Published Language)

발행된 언어는 하류의 컨텍스트가 상류 컨텍스트가 제공하는 기능을 사용하게 하기 위한 간단한 사용과 번역을 가능케 하는 문서화된 정보 교환 언어이다. XML, JSON 스키마로 표현될 수 있고 주로 OHS과 조합하여 사용된다.

5.4 이벤트 스토밍을 통한 마이크로서비스 도출

이 문단에서는 알베르토 브란돌리니가 고안한 DDD 설계를 하는 방법인 이벤트 스토밍 방법을 소개한다.

서적에서는 모든 이해관계자가 한 곳에 모여 지식을 공유하여 도메인 모델링을 하는 방법을 알려주지만 여기에는 설계에 도움 되는 절차만 요약한다.

이벤트 스토밍은 브레인 스토밍과 비슷하게 도메인에서 비즈니스 이벤트를 시각화하여 도출해내는 방법이다.

이벤트 스토밍은 현실세계의 도메인을 다양한 의미의 스티커로 표현한다.

1. 도메인 이벤트 찾기

시간의 흐름에 따라 시스템의 동작을 의미하는 도메인 이벤트를 도출한다. 데이터나 데이터의 구조가 아니라 비즈니스 흐름에서 발생한 이벤트에 초점을 둬야 한다.

이벤트는 왼쪽에서 오른쪽으로 시간 흐름순으로 붙이되 연쇄적으로 발생하는 이벤트는 바로 옆에 붙인다. 같은 시점에 족너에 따라 대체적으로 발생할 수 있는 이벤트는 세로로 아래쪽에 같은 선상에 붙인다.

도메인 이벤트는 비즈니스의 어떤 상태를 CUD 하는 요소이다. 시스템의 UI가 아니라 비즈니스 흐름의 상태 변경 과정을 떠올리면서 수행한다.

아래는 쇼핑몰 예제의 도메인 이벤트가 도출된 모습이다. 회원가입 후 상품등록, 주문, 배송이 이뤄지는 이벤트들이 파악됐다.

2. 외부 시스템 도출

외부 시스템과 연계를 통해 업무 흐름이 진행될때 핑크색 스티커에 프로세스 이름을 적어 이벤트 오른쪽 상단에 붙이고 화살표로 외부시스템을 호출한다는 것을 나타낸다. 시스템 구현 범위가 아니라도 기능에 연결된 시스템은 모두 표현한다.

3. 커맨드 찾기

도메인 이벤트를 동작하게 하는 커맨드를 찾는다. 커맨드명은 도메인 이벤트를 동작하게 하는 것으로 동사(명령형) 형태로 작성한다. 하나의 커맨드에 의해 여러개의 이벤트가 동시 또는 연속해서 발생할 수 있으며, 조건에 따라 하나의 커맨드에 여러개의 다른 이벤트가 발생할 수 있음에 유의한다.

아래 그림에서는 파란색 스티커의 커맨드가 orange 스티커인 이벤트를 트리거한다고 표현된 것이다.

4. 핫스팟 찾기

이벤트 스토밍 과정에서 의문 사항이 생기거나 참여하는 사람들이 결정하기 힘든 사항, 다른 부서나 외부에 문의할 필요가 있는 사항은 핫스팟이라는 보라색 스티커를 붙인다.

5. 액터 찾기

커맨드를 찾고 나면 커맨드를 실행하는 액터(actor) 를 도출한다. user, organization, role 등을 의미하며 추상적으로 식별하지 않고 구체적을 역할을 도출해야 한다. 회원이나 관리자 같은 개념보다는 판매자, 구매자, 상품관리자, 배송관리자 와 같이 구체적으로 도출한다. 액터를 도출하면서 이전에 식별되지 않은 커맨드나 도메인 이벤트가 추가로 나오면 추가할 수 있다.

액터는 노란색의 작은 스티커를 사용해 커맨드 왼쪽 아래에 붙여 액터가 커맨드를 조작한다는 것을 명시적으로 표현한다.

액터를 도출하면 다음과 같은 문장으로 커맨드, 이벤트를 검토할 수 있다.

  • 판매자가 상품을 등록하면 [상품등록됨] 이라는 이벤트가 발생함으로서 시스템이 동작한다.
  • 구매자가 상품 주문을 취소하면 [상품주문취소됨] 이벤트가 발생하고, 이어서 주문 취소된 상품의 재고를 변경하는 [재고변경됨] 이벤트가 발생하여 시스템이 동작한다.

이 문장이 자연스럽지 않다면 커맨드와 도메인 이벤트를 변경하거나 새로 도출해야 한다.

이 단계까지 완료되면 전체 시스템의 큰 그림을 시각화하여 생각하지 못했던 것들을 알 수 있게 되며, 쉽게 공유된다.

6. 애그리거트 정의하기

애그리거트는 커맨드와 도메인 이벤트가 영향을 주는 데이터 요소로 노란색 포스트잇에 애그리거트 이름을 써서 커맨드와 도메인 이벤트 사이 상단에 겹쳐 붙인다.

액터와 마찬가지로 구체적인 표현으로 도출하면 좋은데 이유는 애그리거트를 구체적으로 식별할수록 컨텍스트의 경계를 식별하는 데 유용하기 때문이다.

7. 바운디드 컨텍스트 정의하기

애그리거트 이름에서 이 요소들이 비슷한 지 확인할 수 있다. 이름이 같거나 유사한 애그리거트를 완전히 다른 애그리거트와 구분해서 경계를 그린다. 이전에 식별했던 도메인 이벤트, 커맨드, 액터, 애그리거트를 모두 고려해 경계를 식별한다.

식별한 경계는 바운디드 컨텍스트가 된다. 각 바운디드 컨텍스트에는 이름을 붙이는데 보통 컨텍스트 내부 애그리트 이름으로 한다. 컨텍스트 내에 복수개의 애그리거트가 있는 경우 전체를 대표하는 이름으로 선정한다.

애그리거트가 식별된 결과를 보면 동일한 이름의 애그리거트가 멀리 떨어져 있을 수 있는데 이 경우 큰 애그리거트가 있는 곳으로 옮길수 있다.

다시 말해 모든 관련 요소들을 애그리거트 중심으로 모듈화해야 한다는 것을 의미한다.

아래 예제는

  1. 상품, 카테고리 애그리거트를 상품 바운디드 컨텍스트로 묶었다.
  2. 구매 컨텍스트에 [주문], [결제], [주문아이템]이 애그리거트로 식별됐지만 주문할 아이템을 검색한 후 주문에서 결제까지 하나의 흐름으로 이어지기에 분리하지 않고 [구매] 컨텍스트로 묶어서 정의했다.

정책 도출 및 연관관계 정의
정책은 이벤트 뒤에 따라오는 반응적인 비즈니스 로직으로 해당 커맨드를 동작하게 한다.

WHEN [도메인 이벤트] THEN EXECUTE [커맨드]

정책은 도메인 이벤트와 커맨드 사이에 존재한다 정책이 호출하는 커맨드는 제한되지 않고 어떤 컨텍스트에던 있을 수 있다. 그러므로 정책을 도출하면 다른 바운디드 컨텍스트와의 관계도 식별할 수 있다.

아래 예제에서는 구매가 완료 or 취소 되는 경우 재고수량이 변경돼야 한다는 정책이 도출되었다. 아래 순서대로 움직인다.

  1. 구매 Context에서 [구입완료] [결제취소됨] 이벤트
  2. [재고변경] 정책이 실행
  3. [상품재고수정] 커맨드가 호출
  4. [상품재고수정됨] 이벤트가 발생.
  1. 컨텍스트 매핑하기

이 단계 이전까지 완료하면 대략적인 바운디드 컨텍스트가 식별되어 관계가 파악이 되므로 이 단계에서는 이 관계들을 컨텍스트 맵으로 표현해야 한다.

관계를 확인할 때는 다음을 고려한다.

  1. 호출 관계의 방향을 고려한다.
  2. 호출 방식이 동기, 비동기 중 어떤것이 필요한지 판단한다.
    • 호출 방식은 데이터의 일관성 측면과 컨텍스트의 가용성 측면을 고려해서 선택한다.
    • 컨텍스트간 일관된 데이터가 필요하면 동기 호출(실선)로 표현한다.
    • 그 외의 경우는 eventual consistency로 충분하므로 비동기 호출(점선)로 표현한다.

예제에서는 아래와 같이 2가지 경우로 처리한다.

  • 구매 -> 상품, 회원에 동기방식으로 연동
  • 구매 -> 배송 비동기 방식으로 연동

이렇게 도출한 바운디드 컨텍스트는 마이크로서비스 후보가 된다.

최종적으로 마이크로서비스로 정의하려면 다음과 같은 질문을 고려해 판단해본다.

  • 비즈니스 프로세스를 수행하기 위한 하나의 맥락의 단위로 구분될 수 있는가? ( 비즈니스 관점 )
  • 마이크로서비스별로 분리된 데이터를 정의할 수 있는가? ( 데이터 관점 )
  • 하나의 팀이 독립적으로 운영 가능한 단위인가? (운영 조직 관점 )
  • 독립적으로 배포 가능한 단위인가? (배포 관점)
  • 변경 시 영향을 받는 마이크로서비스가 존재하는가? ( 변경 영향도 관점)
  • 도입을 통한 기대효과를 충분히 활용할 수 있는가? ( 클라우드/MSA 도입 목적 관점)

이를 만족한다면 바운디드 컨텍스트는 마이크로서비스로 식별하기에 충분하며 그렇지 않다면 각 질문의 결과에 따라 바운디드 컨텍스트를 분리하거나 통합해서 마이크로서비스로 식별한다.

이벤트 스토밍 결과는 참여자간 공유하여 보완하고 리뷰될 수 있게 해야 한다.

결과는 헥사고날 아키텍처에서 아래와 같이 매핑되어 구현하면 된다.

커맨드는 인바운드 어댑터 중 REST API가 되고 애그리거트는 내부 영역인 도메인 모델이 되며 도메인 이벤트, 외부 시스템 연결은 외부영역의 아웃바웃드 어댑터로 처리할 수 있다.

5.6 도메인 모델링

마이크로서비스 내부 영역은 서비스 특성에 맞게 다양하게 수립할 수 있다. 도메인 모델 중심의 구조 또는 간단한 경우 트랜잭션 스크립트 모델로 작성해도 무방하다. (어떤 구조를 사용할 지는 팀의 역량 수준도 고려해야 한다.)

5.6.1 DDD의 전술적 설계

일반적인 객체 모델링 방식은 자유도가 높아 문제 영역을 파고들수록 복잡한 계측 구조를 만들게 될 가능성이 높은데 DDD의 전술적 설계 방법은 객체들의 역할에 따른 유형을 정의하는 방법을 제공한다.

아래 요소들은 도메인 모델을 구성하는 객체 구성 요소를 설명한다.

Entity
다른 엔티티와 구별할 수 있는 식별자를 가진 도메인의 실체 개념을 표현한다. 식별자는 고유하지만 속성 및 상태는 mutable 하다. 도메인에서 개별성이 있는 개념을 엔티티로 식별하며, 고유 식별자와 변화 가능성이 엔티티가 VO를 구분하는 차이점이다.

구매 엔티티는 식별자를 가졌으므로 엔티티이며 구매자, 구매품 등은 식별자가 없으므로 Value Object이다. VO는 J2EE에서 사용되는 Value Object와 다른 개념으로 J2EE에서 사용된 VO는 DTO로 부르는 것이 적절하다.

Value Object
VO는 속성이 개별적으로 변화하지 않는 개념적 완전성을 모델링한다 개념적 완전성은 값 객체를 구성하는 하나 이상의 특성들이 서로 연관되어 전체 의미를 이루는 것을 말한다. VO는 개별 속성이 개별적으로 수정되지 않고 전체 객체가 한번에 생성되거나 삭제되는 객체다.

엔티티와 같이 식별자의 차이에 따라 구별되지 않고 ducktyping처럼 모든 속성이 같은 값을 가졌느냐에 따라 동일함이 결정된다.

Standard Type
표준 타입은 대상의 타입을 나타내는 서술적 객체다. 엔티티나 값 객체의 속성을 구분하는 용도로 사용한다.

예를 들어 VO가 전화번호이면 이 전화번호가 집 전화인지, cell phone인지 구분할 때 standard type을 사용할 수 있다. 일반적으로 데이터베이스 모델링에서 코드 값으로 사용된 것과 같은 역할이다. 일반적으로 java 에선 enum으로 정의한다.

Aggregate
엔티티와 값, 표준 타입 객체들로 도메인을 모델링하면 객체간 계층 구조가 만들어진다.

이 객체들을 묶어서 대표하는 객체가 애그리거트다. 애그리거트 하위 객체는 비즈니스 의존 관계를 갖고 있으므로 비즈니스 정합성을 일치시켜야 한다. 그래서 애그리거트는 트랜잭션의 기본 단위가 되어 하위 객체들의 정합성을 일치시켜준다.

애그리거트는 내부 엔티티 중 가장 상위 객체를 애그리거트 루트로 정한다. 이 루트 객체는 다른 엔티티와 VO를 대표하여 다른 컨텍스트와 연계할 때 해당 객체를 통해 통신하게 된다.

하나의 컨텍스트 안에 여러개의 애그리거트가 있고 다른 애그리거트르 참조(reference) 해야 한다면 직접 객체 인스턴스르 갖고 있지 않고 primary 키 같은 식별자를 통해 참조한다. 직접 참조하는 경우의 다음과 같은 단점이 있기 때문이다.

  • 애그리거트 단위의 트랜잭션 처리 어려움
  • 의존관계 복잡성

애그리거트 사이의 일관성은 domain event를 통한 eventual consistency로 유지한다.

도메인 서비스
도메인의 비즈니스 로직 처리가 특정 엔티티나 값 객체에 속하지 않을 때 단독 객체를 만들어서 처리하는 것을 말한다. 도메인 서비스는 상태가 없이 행위만 존재하여 다른 엔티티나 값 객체가 가진 상태값을 통해 로직을 처리한다.

도메인 이벤트
이벤트 스토밍의 결과로 도출된 도메인 이벤트 개념의 구현체로서 서비스간 정합성을 일치시키기 위해 단위 애그리거트의 주요 상태값을 담아 전달되도록 모델링한다.
