친절한 SQL 튜닝 정리 - 5

친절한 SQL 튜닝 정리 - 5

이 문서는 조시형의 저서 친절한 SQL 튜닝을 개인적으로 참조하기 위해 요약,정리한 것으로 분산된 내용이 많아 책 없이 읽을 수 없는 문서입니다.



5장 소트 튜닝

SORT UNIQUE

subquery unnesting 후 subquery가 유일성이 없다면 ( M쪽 집합이거나 1쪽 집합이더라도 Unique 인덱스가 없으면 ) 다시 말해 상대 조인 쿼리 블록보다 다수이면 메인 쿼리와 조인하기 전에 중복 레코드를 제거하는 절차가 필요하다.
이 절차를 Sort Unique 오퍼레이션이라고 한다.

서브쿼리의 유일성이 보장되면 Sort Unique 오퍼레이션이 생략된다.

Sort Unique 오퍼레이션은 Union, Minus, Intersect 같은 집합 연산자를 사용할 때 나타나는 경우가 많다.
또한 Distinct 연산자를 사용해도 Sort Unique 오퍼레이션이 나타난다.

Sort Join - 소트 머지 조인 수행시 사용되는 오퍼레이션

Window Sort - 윈도우 함수 사용시 사용하는 오퍼레이션

5.2 소트 발생하지 않도록 SQL 작성하기

5.2.1 Union VS Union All

Union은 중복을 없애고 표시하는 것이 다르다.
다시 말해 Sort 연산을 수행하므로 합해지는 각 대상 쿼리에서 중복을 없애고 Union All을 사용하는 것이 좋다.

5.2.2 Exists

Distinct 대신 Exist를 사용하면 좋은 점이 많다

Distinct 연산자는 데이터를 모두 읽고 중복제거를 해야 하고, 부분범위 처리가 불가하다.
그러나 Exists는 데이터 존재 여부만 확인하기에 데이터를 전부 읽지 않고 부분범위 처리도 가능하다.

5.2.3 조인 방식 변경

조건절에 인덱스 컬럼 명시 순서에 따라 소트 연산이 생략이 가능하지만 인덱스를 바꾸거나 생성할 수 없을 때에는 조인 방법을 바꿔서 소트 연산을 없앨 수도 있다.

5.3 인덱스를 사용한 소트 연산 생략
SQL에 order by 문장이 있어도 조건절에 인덱스 컬럼 순서대로 명시되어 있다면 소트 연산이 생략이 가능하다.

5.3.2 Top N 쿼리
Top N 쿼리는 DBCP를 사용하는 3-Tier 환경에서도 부분범위 처리를 가능하게 해 준다.

5.3.3 인덱스를 이용한 최솟값/최댓값 구하기
최솟값/최댓값을 구할 때 전체값을 읽어서 일반적으로 Sort Aggregate 오퍼레이션이 나타나지만 인덱스를 이용하면 전체 데이터 읽지 않고도 구할 수 있다.(오라클 8i 이상)

이렇게 하려면 조건절 컬럼과 MIN/MAX 함수의 매개변수 컬럼이 모두 인덱스에 포함되어 있어야 한다.(테이블 액세스 없음)

5.3.4 TOP N 쿼리를 이용한 최솟값 최댓값 구하기
위 두개를 합친것으로 FROM 절의 서브쿼리에서 ORDER BY DESC를 하고 메인쿼리블록의 조건절을 ROWNUM <= 1 로 부여한다.
그러면 COUNT (STOP KEY) 가 실행계획에 등장하는데 이것은 쿼리가 어디에서 멈출지면 최솟값.최댓값인지 엔진에서 패턴으로 인식했다는 뜻이다.

이력조회

책에서 장비 테이블과 그 장비의 상태 변경이력 테이블을 보여준다. 장비 테이블엔 최종변경일자와 상태코드가 있고 상태변경이력은 상태코드가 변경되었을 때의 이력이 기록되는 곳이다 장비와 상태변경이력 테이블을 일대다 관계이다.

첫번째 사례는 단순 이력조회다.

이력테이블에서 MAX를 사용한 스칼라 서브쿼리를 통해 최종변경일자를 알아내고 조회한다.
Top N StopKey 알고리즘이 실행계획에서 작동한 것이 보인다.(5.3.4.1 쿼리)

두번째 사례는 단순 이력조회 쿼리를 여러번 생성한 것으로 FROM 절에 필요한 필드가 text concat 된 인라인 뷰를 명시하였고 메인쿼리에서는 그것을 substr(분해) 해서 사용했다. (5.3.4.2 쿼리)

이 컬럼의 문제는 인라인 뷰에서 인덱스 컬럼을 가공해 First Row StopKey 알고리즘이 동작하지 않는다는 것이다.

그리고 First Row StopKey가 동작하도록 개선된 쿼리(5.3.4.3 쿼리)에서는 단순이력조회처럼 인라인뷰를 쓰지 않고 여러개의 스칼라 서브쿼리를 명시하여 사용하였다.
이 쿼리에서는 First Row StopKey가 동작하지만 상태변경이력을 여러번 조회하는 비효율이 존재한다.

이것은 상태변경이력 테이블에 데이터가 많을 때 문제가 되는데 이것을 해결하려면 (5.3.4.2 쿼리) 의 인라인뷰에서 INDEX_DESC 힌트와 ROWNUM <= 1 조건절을 사용하면 COUNT STOPKEY 알고리즘이 작동하게 만들어 해결할 수 있다(TOP N 쿼리를 이용한 최솟값 최댓값 구하기 )

(5.3.4.3 쿼리)의 두번째 개선은 11g/12c에서 사용하는 인라인 뷰에서 메인쿼리의 값을 사용하는 Predicate Pushing 쿼리변환을 사용하는 사례를 보여준다.

이런 이력 테이블을 배치로 처리할때는 STOP KEY 알고리즘을 고려하지 말고 Window 함수를 사용하는 것이 효과적이다.

선분이력

시작일자와 종료일자 컬럼이 있는 이력테이블을 사용하는 것으로 쿼리도 간단해지고 성능 측면에서 이점도 있다.

5.3.5 Sort Group By 생략

Group By 연산시 부분범위 처리가 가능하게 인덱스를 활용하여 Sort Group By 연산을 생략하는 것이다.

5.4 Sort Area를 적게 사용하도록 SQL 작성

소트 연산이 불가피할 때 Sort Area를 사용하는 방법은 Sort Area를 크게 늘리는 것이 아니라 적게 사용하는 방법을 찾는 것이 순서이다.

5.4.1 소트 데이터 줄이기

첫번째 사례는

from 절에서 인라인 뷰를 사용하지 않은 쿼리(5.4.1.1) 와 인라인 뷰를 사용한 쿼리(5.4.1.2) 를 비교한다 메인 쿼리에는 필드한 컬럼이 text concat 되어 있다.

차이는 인라인 뷰 안에서 먼저 concanated text가 가공될 때 발생한다.

(5.4.1.1)쿼리는 가공된 컬럼의 결과집합이 Sort Area에 저장되고
(5.4.1.2)쿼리는 인라인 뷰 안에서 정렬된 후에 가공된다.
(5.4.1.2)가 가공되지 않은 필드를 사용해 정렬하므로 더 적게 사용한다.

두번째 사례는 SELECT-LIST만 다른 동일한 쿼리에서 모든(*) 컬럼을 출력하는 쿼리와 일부만 출력하는 쿼리이다.
일부만 출력하는 쿼리가 더 적은 쿼리만 사용한다.

5.4.2 Top N 쿼리의 소트 부하 경감 원리

여러개의 컬럼 업데이트시

  1. 스칼라 서브쿼리로 각 컬럼 업데이트( 같은 테이블이 여러개이므로 비효율적 )

  2. 여러개 컬럼을 인라인 뷰에서 업데이트 하는 경우( 설명하기 어렵지만 비효율 있음 )

  3. 수정 가능 조인 뷰( updatable join view )
    일단 쿼리의 모양에서 다른 점은 1,2 번은 update 테이블을 지정할 때 일반 테이블만 지정하고 set 절에서 서브쿼리가 있지만 수정가능 조인 뷰는 업데이트 대상 테이블에 인라인 뷰가 위치한다.

    이때 업데이트가 적용되는 테이블은 테이블 중 M쪽 테이블이다.
    1쪽 테이블은 unique 하게 식별 가능해야 하며, 이 테이블을 오라클에서는 키 비보존 테이블(non key-preserved table) 이라고 부른다.
    M쪽 테이블은 1쪽 테이블에 1:M쪽으로 대응되는 다수쪽 테이블이며 오라클에서는 키 보존 테이블(key-preserved table) 이라고 부른다.
    키 보존 테이블은 조인된 뷰에 자신의 row id를 제공하는 테이블을 말한다.
